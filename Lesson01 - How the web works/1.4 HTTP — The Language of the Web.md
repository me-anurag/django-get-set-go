# 1.4 HTTP — The Language of the Web

> HTTP is the protocol that makes the web work. Django speaks it natively. Understanding it is non-negotiable.

---

## What is a Protocol?

Before HTTP, a definition: a **protocol** is a set of rules that two parties agree to follow so they can communicate reliably.

When two people call each other on the phone, there's an implicit protocol: one person calls, the other answers, you take turns speaking, you say goodbye before hanging up. Without these rules, communication breaks down.

On the internet, computers need formal, explicit protocols — agreed-upon formats and rules for how data is structured, sent, and interpreted.

**HTTP (HyperText Transfer Protocol)** is the protocol the web uses. It defines:
- How a client formats a request
- How a server formats a response
- What the different components of each mean

---

## A Brief History

| Year | Event |
|------|-------|
| **1989** | Tim Berners-Lee proposes the World Wide Web at CERN |
| **1991** | HTTP/0.9 — single-line requests, HTML-only responses |
| **1996** | HTTP/1.0 — introduced headers, status codes, multiple content types |
| **1997** | HTTP/1.1 — persistent connections, chunked transfer; still widely used |
| **2015** | HTTP/2 — binary format, multiplexing, header compression |
| **2022** | HTTP/3 — built on QUIC instead of TCP, faster connection setup |

You'll work primarily in an HTTP/1.1 mental model when learning Django — the concepts (methods, headers, status codes, request/response structure) are identical across versions. The later versions are optimizations, not redesigns.

## HTTP is a Text Protocol

HTTP/1.1 requests and responses are **plain text**. You can read them. You can write them by hand. This is one of HTTP's most important characteristics — it makes debugging, learning, and tooling straightforward.

A raw HTTP request looks like this:

```
GET /articles/django-intro/ HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Accept: text/html
Accept-Language: en-US
Connection: keep-alive
```

A raw HTTP response looks like this:

```
HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Content-Length: 3421
Date: Sat, 30 May 2026 10:00:00 GMT

<!doctype html>
<html>
  <head><title>Django Intro</title></head>
  ...
</html>
```

Everything you see there is human-readable text. There is no magic.

## The Structure of an HTTP Request

Every HTTP request has the same structure:

```
[Method] [Path] [HTTP Version]    ← Request Line
[Header]: [Value]                 ← Headers (one per line)
[Header]: [Value]
                                  ← Blank line (separates headers from body)
[Body]                            ← Optional body (present in POST, PUT, etc.)
```

### The Request Line

```
GET /articles/django-intro/ HTTP/1.1
 │         │                    │
Method   Path              HTTP version
```

- **Method** — what action you want to perform
- **Path** — which resource you're acting on
- **HTTP version** — which version of the protocol to use

### Headers

Headers are key-value pairs that carry metadata about the request:

```
Host: www.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0)
Accept: text/html,application/xhtml+xml
Accept-Language: en-US,en;q=0.9
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
Content-Type: application/json
```

| Header | What it tells the server |
|--------|--------------------------|
| `Host` | Which domain this request is for |
| `User-Agent` | What browser/client is making the request |
| `Accept` | What content types the client can handle |
| `Accept-Language` | Preferred language(s) |
| `Authorization` | Credentials (for protected endpoints) |
| `Content-Type` | The format of the request body (if present) |
| `Cookie` | Stored cookies sent back to the server |

In Django, all request headers are accessible via `request.headers` or `request.META`.

### The Body

The body is optional. It carries data being sent *to* the server — most commonly in POST, PUT, or PATCH requests.

```
POST /api/articles/ HTTP/1.1
Host: www.example.com
Content-Type: application/json
Content-Length: 58

{"title": "Django for Beginners", "author": "Noting"}
```

The blank line between the headers and the body is mandatory — it's how the parser knows where headers end and the body begins.

## HTTP Methods

The **method** tells the server what kind of action the client wants to perform. Each method has a specific, well-defined semantic meaning.

| Method | Meaning | Has Body? | Idempotent? |
|--------|---------|-----------|-------------|
| **GET** | Retrieve a resource | No | Yes |
| **POST** | Submit data, create a resource | Yes | No |
| **PUT** | Replace a resource entirely | Yes | Yes |
| **PATCH** | Update part of a resource | Yes | No |
| **DELETE** | Delete a resource | No | Yes |
| **HEAD** | Like GET but return headers only | No | Yes |
| **OPTIONS** | Return what methods the server allows | No | Yes |

**Idempotent** means you can send the same request multiple times and get the same result. `GET /articles/1/` called ten times returns the same article ten times — no side effects. `POST /articles/` called ten times creates ten articles — clearly not idempotent.

### How Browsers Use Methods

A plain HTML browser naturally uses only two methods:
- Typing a URL or clicking a link → **GET**
- Submitting an HTML form → **GET** or **POST** (whichever `method=` attribute says)

For PUT, PATCH, and DELETE, browsers need JavaScript (e.g., `fetch()`) or form method overriding. This is why Django REST Framework exists, and why you'll eventually use `fetch()` or Axios in frontend work.

### Methods in Django

In a Django view, you access the method via `request.method`:

```python
def article_view(request, pk):
    if request.method == 'GET':
        # Return the article
        ...
    elif request.method == 'POST':
        # Create or update something
        ...
```

Django also provides decorators to restrict which methods a view accepts:

```python
from django.views.decorators.http import require_GET, require_POST

@require_POST
def create_article(request):
    # Only accepts POST; returns 405 for anything else
    ...
```

## The Structure of an HTTP Response

Every HTTP response has the same structure:

```
[HTTP Version] [Status Code] [Reason Phrase]   ← Status Line
[Header]: [Value]                               ← Headers
[Header]: [Value]
                                                ← Blank line
[Body]                                          ← Optional body
```

### The Status Line

```
HTTP/1.1 200 OK
    │      │   │
Version  Code  Reason phrase
```

The **status code** is the machine-readable part. The **reason phrase** is the human-readable label.

### Status Code Families

| Range | Family | Meaning |
|-------|--------|---------|
| **1xx** | Informational | Request received, continuing |
| **2xx** | Success | Request understood and fulfilled |
| **3xx** | Redirection | Further action needed |
| **4xx** | Client Error | The request has a problem |
| **5xx** | Server Error | The server failed to fulfill a valid request |

### Status Codes You'll Use Daily as a Django Developer

| Code | Name | When you'll see it |
|------|------|--------------------|
| `200` | OK | Successful GET request |
| `201` | Created | Successful POST that created a resource |
| `204` | No Content | Successful DELETE (nothing to return) |
| `301` | Moved Permanently | Permanent redirect (e.g., HTTP → HTTPS) |
| `302` | Found | Temporary redirect (e.g., after form submission) |
| `400` | Bad Request | Invalid data submitted |
| `401` | Unauthorized | Not authenticated |
| `403` | Forbidden | Authenticated but not permitted |
| `404` | Not Found | Resource doesn't exist |
| `405` | Method Not Allowed | Wrong method for this endpoint |
| `422` | Unprocessable Entity | Validation error (common in APIs) |
| `500` | Internal Server Error | Your code crashed |

In Django, you return status codes in your views:

```python
from django.http import HttpResponse, JsonResponse

# Implicit 200
return HttpResponse("Hello")

# Explicit status code
return HttpResponse("Created", status=201)
return JsonResponse({"error": "not found"}, status=404)

# Django shortcuts
from django.shortcuts import get_object_or_404
article = get_object_or_404(Article, pk=pk)  # Raises 404 if not found
```

## HTTP is Stateless

This is important enough to state clearly and completely:

**HTTP has no memory.** Every request is independent. The server does not remember previous requests from the same client.

After completing a request-response cycle:
- The server does not remember the client
- The client does not maintain a live connection
- The next request starts completely fresh

This is by design. Statelessness makes HTTP simple, scalable, and predictable. A server can handle millions of clients without tracking their state.

But applications need state. You need to know if a user is logged in. You need to maintain a shopping cart. You need to remember user preferences.

The solution is to **put state somewhere else** and attach it to each request:

| Mechanism | How it works |
|-----------|-------------|
| **Cookies** | Small pieces of data stored in the browser and sent with every request |
| **Sessions** | Data stored on the server, referenced by a session ID sent in a cookie |
| **Tokens (JWT)** | Self-contained tokens sent in request headers |
| **Query parameters** | State encoded in the URL itself |

Django's authentication system, session framework, and CSRF protection are all built on this foundation — they exist specifically to manage state in a stateless protocol.

## HTTP vs HTTPS

**HTTPS** is HTTP with **TLS encryption** layered on top.

The protocol itself — methods, headers, status codes, request/response structure — is identical. The difference is that all data is encrypted in transit:

| | HTTP | HTTPS |
|-|------|-------|
| **Port** | 80 | 443 |
| **Encryption** | None | TLS (SSL) |
| **Data in transit** | Readable by anyone | Encrypted |
| **Authentication** | None | Server identity verified via certificate |
| **Django requirement** | Dev only | Production always |

In modern web development, HTTPS is mandatory in production. Browsers mark HTTP sites as "Not Secure." Search engines penalize them. Django's security checklist will flag it.

Django doesn't handle TLS directly — that's the job of Nginx or a load balancer in front of it. Django trusts that the infrastructure handles encryption and works with the decrypted request.

## Inspecting HTTP in Practice

The best way to understand HTTP is to watch it happen.

In Chrome or Firefox:
1. Open any website
2. Press `F12` → **Network** tab
3. Reload the page
4. Click any request

You'll see:
- The exact **URL and method**
- All **request headers** sent by your browser
- The **status code** returned
- All **response headers**
- The full **response body**

This is real HTTP traffic. Everything in this lesson, live.

As a Django developer, the Network tab is one of your most important debugging tools. You'll use it to verify that forms are submitting correctly, that API endpoints are returning the right status codes, that headers are set properly, and that redirects are going where they should.

## Key Terms

| Term | Definition |
|------|------------|
| **HTTP** | HyperText Transfer Protocol — the protocol for web communication |
| **HTTPS** | HTTP with TLS encryption |
| **Protocol** | A set of agreed-upon rules for communication |
| **Method** | The action a request wants to perform (GET, POST, PUT, PATCH, DELETE) |
| **Header** | A key-value metadata pair in a request or response |
| **Status Code** | A 3-digit number indicating the outcome of a request |
| **Stateless** | Each request is independent; no built-in memory between requests |
| **TLS** | Transport Layer Security — the encryption layer in HTTPS |
| **Idempotent** | An operation that produces the same result regardless of how many times it's called |

---

## Summary

- **HTTP** is the text-based protocol that governs all web communication
- Requests consist of: a **method**, a **path**, **headers**, and an optional **body**
- The six main methods are **GET, POST, PUT, PATCH, DELETE, HEAD** — each with a specific semantic meaning
- Responses consist of: a **status code**, **headers**, and an optional **body**
- Status codes are grouped: **2xx success, 3xx redirect, 4xx client error, 5xx server error**
- HTTP is **stateless** — cookies, sessions, and tokens are the mechanisms we use to work around this
- **HTTPS** is HTTP with TLS encryption — mandatory in production
- The browser's **Network tab** lets you inspect real HTTP traffic
