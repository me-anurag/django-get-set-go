## Where Django Draws Its Boundary

Django does not handle everything. Understanding what Django is and isn't responsible for is one of the most clarifying things a new Django developer can learn.

### What Django Handles

**URL routing**
Matching the request path to the right Python function. This is `urls.py` — the dispatcher that directs traffic within your application.

**Views**
The Python functions (or classes) that receive a request and return a response. This is your application logic — what to do, what data to fetch, what to return.

**ORM and Database Access**
The abstraction layer over your database. You write Python; Django generates SQL. Queries, inserts, updates, deletes — all via Python objects.

**Templates**
Turning data into HTML. Django's template engine takes a template file and a context dictionary and produces an HTML string.

**Forms**
Parsing, validating, and rendering form data. Django Forms handle the entire lifecycle of user input.

**Authentication**
User accounts, login, logout, password management, sessions, permissions. Django's auth system is built in.

**Middleware**
Cross-cutting concerns that apply to every request and response — CSRF protection, session handling, authentication population, security headers.

**Settings**
Centralised configuration — database, static files, installed apps, security settings, email, caching.

**Admin**
An automatically generated admin interface for managing database records.

### What Django Does Not Handle

**TLS / HTTPS** — handled by Nginx or a load balancer

**Process management** — handled by Gunicorn, systemd, or Docker

**Static file serving in production** — handled by Nginx or a CDN

**Serving at scale (load balancing, CDN)** — infrastructure concern

**Email delivery** — handled by an SMTP server or third-party service (SendGrid, etc.)

**Task queues / async jobs** — handled by Celery + Redis/RabbitMQ

**Real-time messaging (WebSockets)** — handled by Django Channels + ASGI

The cleaner way to say it: **Django is the application layer**. It is surrounded by infrastructure. Good Django development means knowing exactly where your layer begins and ends.

## Django's Core Abstractions, Mapped to the Web

Every major Django concept maps directly to something in the web fundamentals you've learned.

### URLs → HTTP Request Path

The path component of an HTTP request (`/articles/42/`) is what Django's URL router matches against.

```python
# This pattern:
path('articles/<int:pk>/', views.article_detail)

# Matches this request path:
GET /articles/42/ HTTP/1.1
```

URL patterns are Django's way of saying: "when the path looks like this, run this code."

---

### Views → Request Handlers

A Django view is a function that takes an `HttpRequest` and returns an `HttpResponse`. That maps precisely to the request-response cycle.

```python
def article_detail(request, pk):
    # request  ← the complete HTTP request
    # pk       ← captured from the URL path

    article = Article.objects.get(pk=pk)

    # HttpResponse ← the complete HTTP response
    return render(request, 'articles/detail.html', {'article': article})
```

Every HTTP concept you've learned is accessible here:

```python
request.method           # GET, POST, etc.
request.headers          # all request headers
request.GET              # query string parameters
request.POST             # form body data
request.FILES            # uploaded files
request.user             # authenticated user (from session cookie)
request.COOKIES          # all cookies
request.scheme           # 'http' or 'https'
```

And the response you return controls:

```python
HttpResponse(content, status=200)   # status code
response['Content-Type'] = '...'    # response headers
response.set_cookie(...)            # Set-Cookie header
JsonResponse({'key': 'val'})        # Content-Type: application/json
redirect('/somewhere/')             # 302 redirect with Location header
```

---

### Templates → HTTP Response Body

When a view calls `render()`, Django's template engine produces the **HTML that becomes the response body**.

```python
return render(request, 'articles/detail.html', {'article': article})
```

This is equivalent to:

```python
from django.template.loader import render_to_string
html = render_to_string('articles/detail.html', {'article': article})
return HttpResponse(html, content_type='text/html')
```

The template is the body. The `Content-Type: text/html` header tells the browser to parse and render it.

---

### Models → Database → Dynamic Content

Models are Python classes mapped to database tables. When a view queries a model, Django translates it to SQL, sends it to PostgreSQL, receives the results, and wraps them in Python objects.

```python
# This Python expression:
Article.objects.filter(published=True).order_by('-created_at')[:10]

# Becomes this SQL:
# SELECT * FROM articles_article
# WHERE published = TRUE
# ORDER BY created_at DESC
# LIMIT 10;
```

The data that comes back from this query is what makes the response **dynamic** — it's different for different users, different times, different filters. This is the essence of a dynamic web application.

---

### Sessions and Cookies → Stateful HTTP

HTTP is stateless. Django's session framework solves this by:

1. Generating a unique session ID on login
2. Storing session data server-side (in the database, cache, or file system)
3. Setting a `sessionid` cookie in the response
4. On every subsequent request, reading the `sessionid` cookie and loading that user's session
5. `AuthenticationMiddleware` reads the session and populates `request.user`

```
First login:
  POST /login/  →  Django verifies credentials
                →  Creates session in DB: {session_id: 'abc123', user_id: 7}
                →  Set-Cookie: sessionid=abc123

Every subsequent request:
  Cookie: sessionid=abc123
  →  SessionMiddleware loads session → {user_id: 7}
  →  AuthMiddleware loads user → request.user = User(id=7)
  →  Your view can use request.user.is_authenticated, request.user.username, etc.
```

The cookie is the HTTP mechanism. The session system is Django's layer on top. Together they solve statelessness.

---

### CSRF Protection → HTTP Security

Cross-Site Request Forgery (CSRF) is an attack where a malicious website tricks a logged-in user's browser into making a state-changing request to your site.

Because the browser automatically sends cookies with every request to a domain, an attacker's page could submit a form that posts to your Django app with the victim's session cookie.

Django's defence: **every POST request must include a secret token** that the attacker cannot obtain.

```
Django renders the form:
  <input type="hidden" name="csrfmiddlewaretoken" value="random-secret">

Browser submits the form:
  POST /articles/create/
  Cookie: sessionid=abc123; csrftoken=random-secret
  Body: title=...&csrfmiddlewaretoken=random-secret

CsrfViewMiddleware verifies:
  Does the token in the body match the token in the cookie?
  → Yes → allow the request
  → No → 403 Forbidden
```

This works because the attacker's site cannot read your cookies (same-origin policy) and therefore cannot include the correct token in the body.

In templates: `{% csrf_token %}` inserts the hidden field.
In AJAX: read `document.cookie` for `csrftoken` and send in `X-CSRFToken` header.

---

### Middleware → Cross-Cutting HTTP Concerns

Every concept you've learned about HTTP has a corresponding piece of middleware:

| HTTP Concept | Django Middleware |
|-------------|-------------------|
| Sessions (HTTP cookies) | `SessionMiddleware` |
| Authentication (session → user) | `AuthenticationMiddleware` |
| CSRF (token validation) | `CsrfViewMiddleware` |
| HTTPS redirect (301) | `SecurityMiddleware` |
| HSTS (`Strict-Transport-Security`) | `SecurityMiddleware` |
| Clickjacking (`X-Frame-Options`) | `XFrameOptionsMiddleware` |
| MIME sniffing (`X-Content-Type-Options`) | `SecurityMiddleware` |
| URL normalisation (trailing slash) | `CommonMiddleware` |
| GZip compression | `GZipMiddleware` |

Middleware is how Django applies HTTP-level logic to every request without cluttering every view function.

## The Request Object as Your Entry Point

Every view starts with `request`. Now you know exactly what's in it:

```python
def any_view(request, *args, **kwargs):
    #
    # From the HTTP request line:
    request.method        # 'GET' | 'POST' | 'PUT' | 'PATCH' | 'DELETE'
    request.path          # '/articles/42/'
    request.scheme        # 'http' | 'https'
    #
    # From HTTP headers:
    request.headers       # {'Host': '...', 'Accept': '...', ...}
    request.get_host()    # 'myapp.com'
    #
    # From URL query string (?page=2):
    request.GET           # QueryDict({'page': ['2']})
    #
    # From request body (form submission):
    request.POST          # QueryDict({'title': ['Hello'], ...})
    request.body          # raw bytes (for JSON)
    request.FILES         # uploaded files
    #
    # From cookies → session → auth middleware:
    request.COOKIES       # {'sessionid': 'abc123', 'csrftoken': '...'}
    request.session       # {'user_id': 7, ...}
    request.user          # User object or AnonymousUser
    #
    # Convenience:
    request.is_secure()          # True if HTTPS
    request.is_ajax()            # deprecated; use Accept or custom header
    request.get_full_path()      # '/articles/42/?page=2'
```

You are not dealing with raw HTTP strings. You're dealing with a well-structured Python object that Django built from the raw HTTP request. But now you know exactly what that raw request looked like — and that knowledge makes you a better debugger, a better API designer, and a better Django developer.

## The Django Developer's Mental Model

When you work in Django, hold this mental model:

```
Every view is a function:

    HttpRequest → [your Python code] → HttpResponse

The HttpRequest contains the entire HTTP request.
The HttpResponse contains the entire HTTP response.
Middleware wraps both.
The ORM translates Python to SQL between them.
```

That's it. Everything else in Django — forms, templates, class-based views, serializers, signals, admin — is tooling built on top of this single, fundamental pattern.

When something breaks, trace it back to this model:
- What was in the request? (method, headers, body, user)
- What did the view do?
- What is in the response? (status code, headers, body)

The browser's Network tab shows you both ends. Django's debug page shows you the middle. You have full visibility into the entire pipeline.

## Key Terms

| Term | Definition |
|------|------------|
| **Application layer** | Django's position in the stack — receives HTTP requests, returns HTTP responses |
| **MTV** | Model-Template-View — Django's architectural pattern |
| **View** | A Python callable that takes an HttpRequest and returns an HttpResponse |
| **URL dispatcher** | Django's routing system that matches request paths to views |
| **ORM** | Object-Relational Mapper — Django's database abstraction layer |
| **Template** | A file that Django renders into the response body (usually HTML) |
| **Session** | Server-side storage for per-user state, linked to the browser via a cookie |
| **CSRF token** | A secret value in forms that prevents cross-site request forgery |
| **Middleware** | Code that wraps every request and response in Django's pipeline |
| **Static files** | Files served as-is (CSS, JS, images) — handled by Nginx in production |
| **`request`** | The `HttpRequest` object Django passes to every view |
| **`response`** | The `HttpResponse` object every view must return |

## Summary

- Django sits at the **application layer** — between the web server (Nginx/Gunicorn) and the database (PostgreSQL)
- Every Django concept maps directly to an HTTP fundamental:
  - URLs → request path routing
  - Views → request handlers returning responses
  - Templates → response body (HTML)
  - Models → dynamic data from database queries
  - Sessions/cookies → stateful HTTP
  - Middleware → cross-cutting HTTP concerns
  - CSRF protection → HTTP security
- The core Django pattern is: `HttpRequest → [your code] → HttpResponse`
- The browser's Network tab gives you full visibility into the HTTP layer; Django's debug page gives you full visibility into the Python layer
- Everything from lesson 1.1 to 1.9 built the foundation for understanding where Django lives — and why it's designed the way it is
