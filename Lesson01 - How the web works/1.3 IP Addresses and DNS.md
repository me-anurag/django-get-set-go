# 1.3 IP Addresses and DNS

> Every device on the internet has an address. DNS is the system that lets humans avoid memorizing them.

---

## The Problem DNS Solves

Computers on the internet find each other using **IP addresses** — numerical identifiers like `142.250.80.46`. Every server, every device, every machine that communicates on the internet has one.

But humans don't remember numbers well. We remember names.

The **Domain Name System (DNS)** is the bridge between the two. It lets you type `google.com` and have the internet figure out the corresponding IP address automatically.

Without DNS, you'd have to memorize an IP address for every website you visit. DNS makes the web usable for humans.

---

## What is an IP Address?

An **IP address** is a unique numerical label assigned to every device connected to a network.

### IPv4

The format most people recognize:

```
142.250.80.46
```

Four numbers, each between 0 and 255, separated by dots. This gives approximately **4.3 billion** possible addresses — a number that seemed enormous in 1981 and is now nearly exhausted.

### IPv6

The modern replacement for IPv4:

```
2607:f8b0:4004:0c08:0000:0000:0000:003e
```

Eight groups of four hexadecimal digits, separated by colons. This gives approximately **340 undecillion** addresses — enough for every atom on Earth's surface to have its own address many times over.

Both systems coexist on the modern internet. As a Django developer, you'll mostly encounter IPv4 in practice, but IPv6 awareness is increasingly important.

### Public vs Private IP Addresses

| Type | Range | Visible to? |
|------|-------|-------------|
| **Public** | Assigned by ISP | The entire internet |
| **Private** | 192.168.x.x, 10.x.x.x | Only your local network |

Your home router has a **public IP** that the internet sees, and it assigns **private IPs** (like `192.168.1.5`) to devices on your home network. When you run Django locally on `127.0.0.1`, that's a special **loopback address** — it routes back to your own machine and is never sent to the network at all.

```
127.0.0.1   →  "this machine itself" (loopback)
0.0.0.0     →  "all available network interfaces" (used in server binding)
```

When you run `python manage.py runserver`, Django listens on `127.0.0.1:8000` by default — accessible only from your own machine.

## What is DNS?

**DNS (Domain Name System)** is a distributed, hierarchical database that maps domain names to IP addresses.

It is distributed because no single server holds all records. It is hierarchical because it is organized in layers, from broad (`.com`) to specific (`www.yoursite.com`).

### The Hierarchy

```
.                        ← Root (the top of everything)
├── com                  ← Top-Level Domain (TLD)
│   ├── google
│   │   └── www          ← Subdomain
│   └── github
│       └── api
├── org
│   └── wikipedia
└── io
    └── yourapp
```

Every domain name you read from right to left is a path down this tree:

- `www.google.com` → root → `.com` → `google` → `www`

---

## How DNS Resolution Works

When you type `www.google.com` into a browser, here is what happens to resolve it to an IP address:

### Step 1 — Check the Local Cache

The browser first checks its own DNS cache. Did it look up this domain recently? If so, use the cached IP and skip the rest.

### Step 2 — Ask the Recursive Resolver

If not cached, the browser asks a **recursive resolver** — a DNS server configured on your device (usually provided by your ISP or set manually to something like `8.8.8.8` — Google's public DNS).

The recursive resolver does the actual work of finding the answer.

### Step 3 — Resolver Checks Its Cache

The resolver checks its own cache. If it has recently looked up `google.com`, it returns the answer immediately.

### Step 4 — Ask the Root Name Servers

If not cached, the resolver contacts one of the **root name servers**. There are 13 sets of root name servers distributed globally. They don't know the IP for `google.com`, but they know who does — specifically, which name servers are authoritative for `.com`.

### Step 5 — Ask the TLD Name Server

The resolver contacts the `.com` TLD name server. It doesn't know `google.com`'s IP either, but it knows which name servers are authoritative for `google.com` specifically.

### Step 6 — Ask Google's Authoritative Name Server

The resolver contacts Google's authoritative DNS servers — the servers that actually hold Google's DNS records. These return the IP address for `www.google.com`.

### Step 7 — Return and Cache the Answer

The resolver returns the IP to your browser, and both the resolver and the browser cache it for a period defined by the record's **TTL (Time to Live)** — typically anywhere from a few minutes to 24 hours.

```
Browser
   │
   ├── Cache hit? ──Yes──► Use it
   │
   No
   │
   ▼
Recursive Resolver (e.g., 8.8.8.8)
   │
   ├── Cache hit? ──Yes──► Return it
   │
   No
   │
   ▼
Root Name Server (13 clusters globally)
"I don't know, but ask .com TLD servers"
   │
   ▼
.com TLD Name Server
"I don't know, but ask Google's servers"
   │
   ▼
Google's Authoritative Name Server
"www.google.com is 142.250.80.46"
   │
   ▼
Answer returned → cached → used
```

The whole process — if not cached — typically takes 20–120ms.

## DNS Record Types

DNS servers don't just store IP addresses. They store various **record types**:

| Record | Purpose | Example |
|--------|---------|---------|
| **A** | Maps domain to IPv4 address | `google.com → 142.250.80.46` |
| **AAAA** | Maps domain to IPv6 address | `google.com → 2607:f8b0:...` |
| **CNAME** | Alias — maps one name to another | `www.example.com → example.com` |
| **MX** | Mail server for a domain | `example.com → mail.example.com` |
| **TXT** | Arbitrary text — used for verification, SPF | Various |
| **NS** | Name servers for this domain | `example.com → ns1.host.com` |

As a Django developer, you'll most often work with **A records** (pointing your domain to your server's IP) and **CNAME records** (creating aliases like `www` pointing to your root domain or a CDN).

## Ports — the Other Half of the Address

An IP address finds the machine. A **port** identifies which program on that machine should handle the connection.

```
142.250.80.46 : 443
      │            │
  IP address     Port
  (which machine) (which program)
```

Common port assignments:

| Port | Service |
|------|---------|
| 80 | HTTP |
| 443 | HTTPS |
| 22 | SSH |
| 5432 | PostgreSQL |
| 6379 | Redis |
| 8000 | Django dev server (convention) |
| 3306 | MySQL |

When you type `https://google.com`, the browser automatically uses port 443 because HTTPS defaults to it. You only see ports explicitly in URLs when they're non-standard — like `http://localhost:8000`.

## What This Means for Django Deployment

When you deploy a Django application and point a domain at it:

1. You get a **public IP address** from your hosting provider (e.g., `165.22.14.88`)
2. You go to your domain registrar and create an **A record**: `yourapp.com → 165.22.14.88`
3. DNS propagates this record across the internet (up to 48 hours, but usually much faster)
4. Users type `yourapp.com`, DNS resolves it to `165.22.14.88`, their browser connects to your server

This is the mechanical reality behind "going live" with a web application.

## Key Terms

| Term | Definition |
|------|------------|
| **IP Address** | A numerical identifier for a device on a network |
| **IPv4** | The older 32-bit format (e.g., `192.168.1.1`) |
| **IPv6** | The newer 128-bit format supporting far more addresses |
| **DNS** | Domain Name System — maps domain names to IP addresses |
| **Domain** | A human-readable address like `google.com` |
| **TLD** | Top-Level Domain — the last part of a domain (`.com`, `.org`, `.io`) |
| **Recursive Resolver** | The DNS server that does the actual lookup work |
| **Authoritative Name Server** | The DNS server that holds the actual records for a domain |
| **TTL** | Time To Live — how long a DNS record is cached before re-querying |
| **A Record** | DNS record mapping a domain to an IPv4 address |
| **CNAME** | DNS record mapping a domain name to another domain name |
| **Port** | A numbered channel identifying a specific service on a machine |
| **Loopback** | `127.0.0.1` — routes back to the local machine itself |

---

## Summary

- Every device on the internet has an **IP address** — a numeric identifier
- **DNS** maps human-readable domain names to IP addresses
- DNS resolution goes: browser cache → recursive resolver → root servers → TLD servers → authoritative servers
- Results are cached using **TTL** to avoid repeated lookups
- DNS stores different **record types**: A, AAAA, CNAME, MX, TXT, and more
- A **port** specifies which program on a machine should handle a connection
- Deploying Django means pointing a domain's A record at your server's IP
