# 1.7 HTTP Responses in Depth

> Every Django view returns a response. Understanding what that response contains — and what the client does with it — makes you a better view writer.

---

## Anatomy of an HTTP Response

An HTTP response mirrors the structure of a request — three sections separated by a blank line:

```
┌─────────────────────────────────────────────┐
│              STATUS LINE                    │  ← Version, status code, reason phrase
├─────────────────────────────────────────────┤
│                                             │
│               HEADERS                      │  ← Metadata about the response
│                                             │
├─────────────────────────────────────────────┤
│              (blank line)                   │  ← Mandatory separator
├─────────────────────────────────────────────┤
│                                             │
│                 BODY                        │  ← The actual content
│                                             │
└─────────────────────────────────────────────┘
```

## The Status Line

```
HTTP/1.1 200 OK
    │      │   │
    │      │   └── Reason phrase (human-readable label)
    │      └────── Status code (machine-readable)
    └───────────── HTTP version
```

The **status code** is what the client actually acts on. Browsers, API clients, JavaScript's `fetch()`, and all other HTTP clients make decisions based on this number.

---

## Status Codes — Complete Reference for Django Developers

### 2xx — Success

The request was received, understood, and fulfilled.

| Code | Name | When to use |
|------|------|-------------|
| `200` | OK | Successful GET, or successful POST that returns content |
| `201` | Created | Resource was successfully created (POST to create endpoint) |
| `204` | No Content | Success with nothing to return (DELETE, or PUT/PATCH with no body) |
| `206` | Partial Content | Response to a range request (file download resumption) |

**200 vs 201:** When your API creates a resource, return `201 Created` — not `200 OK`. It's more accurate and lets clients know something new was made. The convention is to include the new resource in the response body and a `Location` header pointing to it.

**204 for DELETE:** A successful delete has nothing to return. `204` signals success without a body.

---

### 3xx — Redirection

The client needs to take further action to complete the request — usually following a redirect.

| Code | Name | When to use |
|------|------|-------------|
| `301` | Moved Permanently | Resource has permanently moved to a new URL (SEO: search engines update their index) |
| `302` | Found | Temporary redirect |
| `303` | See Other | After a POST, redirect to a GET — the canonical Post-Redirect-Get pattern |
| `304` | Not Modified | Cached response is still valid; don't send the body again |
| `307` | Temporary Redirect | Like 302, but the method must not change |
| `308` | Permanent Redirect | Like 301, but the method must not change |

**The Post-Redirect-Get pattern** — the most important redirect for Django form handling:

```
User submits form  →  POST /articles/create/
                            │
                       Django processes,
                       creates the article
                            │
                       302/303 redirect
                            │
                       GET /articles/42/
                            │
                       Page renders
```

Why does this matter? Without the redirect, if the user refreshes the "success" page, the browser re-submits the POST — potentially creating a duplicate. The redirect ensures a refresh only re-runs the GET.

In Django:
```python
from django.shortcuts import redirect

def create_article(request):
    if request.method == 'POST':
        form = ArticleForm(request.POST)
        if form.is_valid():
            article = form.save()
            return redirect('article_detail', pk=article.pk)  # 302 redirect
    ...
```

**301 for permanent moves:**
```python
from django.views.generic.base import RedirectView

# In urls.py — permanently redirect old URL to new
path('blog/', RedirectView.as_view(url='/articles/', permanent=True)),
```

### 4xx — Client Errors

The request had a problem on the client's side. The server understood the request but cannot or will not fulfill it as written.

| Code | Name | When to use |
|------|------|-------------|
| `400` | Bad Request | Malformed request, invalid data, missing required fields |
| `401` | Unauthorized | Not authenticated — needs to log in |
| `403` | Forbidden | Authenticated but not permitted — wrong permissions |
| `404` | Not Found | Resource doesn't exist at this URL |
| `405` | Method Not Allowed | Endpoint exists but doesn't support this HTTP method |
| `409` | Conflict | Request conflicts with current state (e.g. duplicate username) |
| `410` | Gone | Resource existed but has been permanently deleted |
| `422` | Unprocessable Entity | Syntactically valid but semantically invalid (validation errors) |
| `429` | Too Many Requests | Rate limit exceeded |

**401 vs 403 — a critical distinction:**

```
401 Unauthorized  →  "I don't know who you are. Please log in."
403 Forbidden     →  "I know who you are. You're not allowed to do this."
```

In Django:
- `@login_required` returns a `302` redirect to login (not a 401) for web views — a common source of confusion
- DRF's `IsAuthenticated` permission returns `401` for unauthenticated API requests

---

### 5xx — Server Errors

Something went wrong on the server. The request was valid, but the server failed to fulfill it.

| Code | Name | When to use |
|------|------|-------------|
| `500` | Internal Server Error | Unhandled exception — your code crashed |
| `502` | Bad Gateway | The proxy received an invalid response from the upstream server |
| `503` | Service Unavailable | Server is temporarily down or overloaded |
| `504` | Gateway Timeout | The upstream server didn't respond in time |

**500 is your most important alarm.** When Django throws an unhandled exception, it returns a 500. In development, Django shows a detailed debug page. In production (`DEBUG=False`), Django shows a generic error page and sends an email to `ADMINS`.

## Response Headers — Reference

### Content Headers

**`Content-Type`** — The format of the response body.

```
Content-Type: text/html; charset=UTF-8
Content-Type: application/json
Content-Type: image/jpeg
Content-Type: text/css
Content-Type: application/javascript
```

This is the most important response header. The browser uses it to decide how to handle the body — render as HTML, parse as JSON, display as an image, etc.

Without a correct `Content-Type`, browsers may misinterpret the body. This is also a security concern — a technique called **MIME sniffing** — which is why the `X-Content-Type-Options: nosniff` header exists.

In Django:
```python
# Default Content-Type is text/html
return HttpResponse("<p>Hello</p>")

# Explicit Content-Type
return HttpResponse('{"key": "value"}', content_type='application/json')

# Django's JsonResponse sets Content-Type automatically
return JsonResponse({'key': 'value'})
```

---

**`Content-Length`** — Size of the body in bytes.

```
Content-Length: 3421
```

Tells the client how many bytes to read. Without it, the client reads until the connection closes. With it, the client knows when the body is complete and can show progress.

---

**`Content-Encoding`** — Compression applied to the body.

```
Content-Encoding: gzip
```

If the server compressed the body (`gzip`, `br`), this header tells the client how to decompress it. The client decompresses transparently before parsing.

---

### Caching Headers

Caching is one of the most impactful performance tools on the web. These headers control how and how long clients and intermediaries cache responses.

**`Cache-Control`** — The primary caching directive.

```
Cache-Control: max-age=3600
Cache-Control: no-cache, no-store, must-revalidate
Cache-Control: public, max-age=86400
Cache-Control: private, max-age=0
```

| Directive | Meaning |
|-----------|---------|
| `max-age=N` | Cache for N seconds |
| `no-cache` | Must revalidate with server before using cache |
| `no-store` | Never cache this response |
| `public` | Can be cached by any cache (CDN, proxy, browser) |
| `private` | Only the user's browser can cache it (not CDNs) |
| `must-revalidate` | Once stale, must check server before serving |

Django provides cache control decorators:
```python
from django.views.decorators.cache import cache_control, never_cache

@cache_control(max_age=3600, public=True)
def static_page(request):
    ...

@never_cache
def user_dashboard(request):
    # Personal data — must never be cached
    ...
```

---

**`ETag`** — A fingerprint of the response content.

```
ETag: "33a64df5"
```

The server generates a hash of the content. The client stores this with the cached response. On the next request, the client sends `If-None-Match: "33a64df5"`. If the content hasn't changed, the server returns `304 Not Modified` with no body — saving bandwidth.

---

**`Last-Modified`** — When the resource was last changed.

```
Last-Modified: Wed, 28 May 2026 10:00:00 GMT
```

Similar to ETag but time-based. The client sends `If-Modified-Since` on subsequent requests. If the content hasn't changed since that time, the server returns `304`.

---

### Security Headers

These headers protect users from common web attacks. Django sets several of them by default via middleware.

**`Set-Cookie`** — Creates a cookie in the browser.

```
Set-Cookie: sessionid=abc123; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=1209600
```

| Attribute | Effect |
|-----------|--------|
| `HttpOnly` | JavaScript cannot read this cookie — prevents XSS theft |
| `Secure` | Only sent over HTTPS |
| `SameSite=Lax` | Only sent on same-site or top-level navigation — CSRF protection |
| `SameSite=Strict` | Only sent on same-site requests |
| `Path=/` | Cookie applies to all paths under this domain |
| `Max-Age=N` | Cookie expires after N seconds |
| `Domain=` | Which domains receive this cookie |

Django sets these attributes on the `sessionid` and `csrftoken` cookies via settings:

```python
# settings.py
SESSION_COOKIE_HTTPONLY = True     # default True
SESSION_COOKIE_SECURE = True       # True in production (HTTPS only)
SESSION_COOKIE_SAMESITE = 'Lax'   # default 'Lax'
CSRF_COOKIE_HTTPONLY = False       # False so JS can read the CSRF token
```

---

**`X-Frame-Options`** — Controls whether this page can be embedded in an iframe.

```
X-Frame-Options: DENY
X-Frame-Options: SAMEORIGIN
```

Prevents **clickjacking** — an attack where your page is hidden inside an iframe on a malicious site. `DENY` blocks all framing. `SAMEORIGIN` allows framing only by the same domain.

Django sets `X-Frame-Options: DENY` via `XFrameOptionsMiddleware`.

---

**`X-Content-Type-Options`** — Prevents MIME sniffing.

```
X-Content-Type-Options: nosniff
```

Tells the browser to trust the `Content-Type` header and not guess the content type. Without this, browsers may "sniff" file content and execute it as a different type — a security risk.

Django sets this via `SecurityMiddleware`.

---

**`Strict-Transport-Security` (HSTS)** — Forces HTTPS.

```
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

Once a browser receives this header, it automatically upgrades all future requests to this domain to HTTPS for `max-age` seconds — even if the user types `http://`. This prevents downgrade attacks.

In Django settings:
```python
SECURE_HSTS_SECONDS = 31536000        # 1 year
SECURE_HSTS_INCLUDE_SUBDOMAINS = True
SECURE_HSTS_PRELOAD = True
```

---

**`Content-Security-Policy` (CSP)** — Controls what resources can load.

```
Content-Security-Policy: default-src 'self'; script-src 'self' cdn.example.com; img-src *
```

Tells the browser which sources are trusted for scripts, styles, images, and other resources. One of the strongest defences against Cross-Site Scripting (XSS) — even if an attacker injects a script tag, the browser won't execute scripts from untrusted sources.

Django doesn't set CSP by default — use the `django-csp` package for this.

---

**`Location`** — Used with 3xx redirects, or after 201 Created.

```
Location: /articles/42/
Location: https://www.example.com/new-page/
```

Tells the client where to go next. The browser follows redirect `Location` headers automatically.

## Django's Response Classes

Django provides a hierarchy of response objects:

```python
from django.http import (
    HttpResponse,
    HttpResponseRedirect,
    HttpResponseNotFound,
    HttpResponseForbidden,
    HttpResponseBadRequest,
    JsonResponse,
    StreamingHttpResponse,
    FileResponse,
)
from django.shortcuts import render, redirect, get_object_or_404
```

| Class | Status | Use case |
|-------|--------|---------|
| `HttpResponse` | Configurable | Base class — any response |
| `JsonResponse` | 200 | JSON API responses |
| `HttpResponseRedirect` | 302 | Redirect to another URL |
| `HttpResponseNotFound` | 404 | Resource not found |
| `HttpResponseForbidden` | 403 | Permission denied |
| `HttpResponseBadRequest` | 400 | Invalid request |
| `StreamingHttpResponse` | 200 | Large responses streamed in chunks |
| `FileResponse` | 200 | File downloads |

**Most common patterns:**

```python
# Render a template
return render(request, 'articles/detail.html', {'article': article})
# render() returns HttpResponse with 200

# Return JSON
return JsonResponse({'status': 'created', 'id': article.pk}, status=201)

# Redirect
return redirect('article_list')          # by URL name
return redirect('/articles/')            # by path
return redirect(article)                 # by model (uses get_absolute_url())

# 404 (raises Http404 exception, caught by Django)
from django.http import Http404
raise Http404("Article not found")

# Or use the shortcut
article = get_object_or_404(Article, pk=pk)

# Set custom headers
response = HttpResponse("Hello")
response['X-Custom-Header'] = 'value'
response['Cache-Control'] = 'max-age=3600'
return response

# Set a cookie
response = HttpResponse("Hello")
response.set_cookie('theme', 'dark', max_age=86400, httponly=True)
return response

# Delete a cookie
response.delete_cookie('theme')
```

## The Full Request-Response in Django

Putting it all together — the path a request takes through Django and back:

```
HTTP Request arrives
        │
        ▼
Django's WSGI handler receives it
        │
        ▼
Middleware stack (request phase)
  SecurityMiddleware
  SessionMiddleware         ← populates request.session
  AuthenticationMiddleware  ← populates request.user
  CsrfViewMiddleware        ← validates CSRF token
  ...
        │
        ▼
URL router matches path → calls view function
        │
        ▼
View function runs
  reads request.method, request.POST, request.user, etc.
  queries the database
  renders a template or builds JSON
  returns HttpResponse
        │
        ▼
Middleware stack (response phase)
  SecurityMiddleware        ← adds security headers
  XFrameOptionsMiddleware   ← adds X-Frame-Options
  ...
        │
        ▼
HTTP Response sent to client
```

Every header you saw in this lesson either comes in with the request and is read by middleware/views, or is added to the response by middleware/views before it leaves.

## Key Terms

| Term | Definition |
|------|------------|
| **Status Line** | First line of a response: HTTP version, status code, reason phrase |
| **Status Code** | 3-digit number indicating the outcome of a request |
| **Content-Type** | Response header declaring the format of the response body |
| **Cache-Control** | Response header controlling caching behaviour |
| **ETag** | A fingerprint of response content, used for cache validation |
| **Set-Cookie** | Response header that creates a cookie in the browser |
| **Location** | Response header specifying the URL for redirects |
| **HSTS** | HTTP Strict Transport Security — forces HTTPS |
| **CSP** | Content Security Policy — controls trusted resource sources |
| **Post-Redirect-Get** | Pattern: POST form → redirect → GET result page |
| **HttpResponse** | Django's class for building HTTP responses |
| **JsonResponse** | Django subclass for JSON responses with correct Content-Type |

---

## Summary

- An HTTP response consists of: **status line**, **headers**, blank separator, **body**
- **2xx** = success, **3xx** = redirect, **4xx** = client error, **5xx** = server error
- **Post-Redirect-Get** is the standard pattern for form submissions — prevents duplicate submissions on refresh
- **Content-Type** tells the browser how to interpret the body
- **Security headers** (`X-Frame-Options`, `X-Content-Type-Options`, HSTS, CSP) protect users from common attacks — most are set by Django middleware automatically
- **`Set-Cookie`** creates browser cookies; `HttpOnly`, `Secure`, and `SameSite` attributes are critical for security
- Django provides `HttpResponse`, `JsonResponse`, `redirect()`, `render()`, and `get_object_or_404()` as the main response-building tools
