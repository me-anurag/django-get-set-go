# 1.6 HTTP Requests in Depth

> A request is a precise, structured message. Every byte has a purpose. Once you can read one, Django's request object stops being a black box.

---

## Anatomy of an HTTP Request

An HTTP request has four parts:

```
┌─────────────────────────────────────────────┐
│              REQUEST LINE                   │  ← Method, path, version
├─────────────────────────────────────────────┤
│                                             │
│               HEADERS                      │  ← Metadata, one per line
│                                             │
├─────────────────────────────────────────────┤
│              (blank line)                   │  ← Mandatory separator
├─────────────────────────────────────────────┤
│                                             │
│                 BODY                        │  ← Optional data payload
│                                             │
└─────────────────────────────────────────────┘
```

Let's examine each section with a real annotated example.

---

## A Complete GET Request

```
GET /articles/?page=2&tag=django HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8
Accept-Language: en-US,en;q=0.9
Accept-Encoding: gzip, deflate, br
Connection: keep-alive
Cookie: sessionid=abc123xyz; csrftoken=defgh456
```

### The Request Line

```
GET /articles/?page=2&tag=django HTTP/1.1
 │         │                │         │
 │         │                │     Protocol version
 │         │         Query string parameters
 │       Path (everything after the domain)
Method
```

The **path** is everything after the domain, including query string. The server uses the path to route the request to the right code.

**Query string parameters** (`?page=2&tag=django`) are key-value pairs appended to the URL with `?`, separated by `&`. They carry optional filtering, pagination, or search data.

In Django, query parameters are accessed via:
```python
page = request.GET.get('page', 1)      # '2'
tag  = request.GET.get('tag', '')      # 'django'
```

Note: `request.GET` is poorly named — it's available on any request method, not just GET. It always refers to **query string parameters**.

---

## A Complete POST Request

```
POST /articles/create/ HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Content-Type: application/x-www-form-urlencoded
Content-Length: 47
Accept: text/html
Cookie: sessionid=abc123xyz; csrftoken=defgh456

title=Hello+World&body=First+post&csrfmiddlewaretoken=defgh456
```

The key difference from a GET request: the **body** — data that wouldn't fit or belong in the URL.

The `Content-Type` header tells the server how to parse the body. The `Content-Length` header tells it how many bytes to read.

---

## Request Headers — Reference

Headers are the metadata layer of every request. Here are the ones you'll encounter most as a Django developer.

### Identification Headers

**`Host`** — Required in HTTP/1.1. Tells the server which domain this request is for.

```
Host: www.example.com
```

Essential because one IP address can serve many domains (virtual hosting). Without `Host`, the server wouldn't know which site to respond with.

In Django: `request.get_host()` returns this value.

---

**`User-Agent`** — Identifies the client software.

```
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/124.0 Safari/537.36
```

Browsers, bots, mobile apps, API clients — they all send different User-Agent strings. Useful for analytics and detecting bots.

In Django: `request.META['HTTP_USER_AGENT']`

---

### Content Negotiation Headers

**`Accept`** — What response formats the client can handle.

```
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8
```

Values are MIME types with optional quality scores (`q=`). The `*/*` means "anything." This lets a server return HTML to browsers and JSON to API clients hitting the same endpoint — a technique called **content negotiation**.

In Django REST Framework, this header drives automatic format selection.

---

**`Accept-Language`** — Preferred languages for the response.

```
Accept-Language: en-US,en;q=0.9,hi;q=0.7
```

Django's internationalisation (i18n) system reads this to serve content in the right language when multiple translations exist.

---

**`Accept-Encoding`** — Compression formats the client supports.

```
Accept-Encoding: gzip, deflate, br
```

If the server supports `gzip`, it can compress the response body before sending — often reducing size by 70–90%. The client decompresses transparently. Django middleware (`GZipMiddleware`) handles this automatically.

---

### Authentication Headers

**`Authorization`** — Credentials for a protected resource.

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Authorization: Basic dXNlcjpwYXNzd29yZA==
Authorization: Token 9944b09199c62bcf9418ad846dd0e4bbdfc6ee4b
```

Three common schemes:
- **Bearer** — a token (JWT or opaque) proving identity
- **Basic** — base64-encoded `username:password` (only safe over HTTPS)
- **Token** — Django REST Framework's built-in token authentication

In Django REST Framework, authentication classes read this header and populate `request.user`.

---

**`Cookie`** — Data stored by the browser from previous responses, sent back automatically.

```
Cookie: sessionid=abc123xyz; csrftoken=defgh456; theme=dark
```

Cookies are sent with every request to the domain that set them — the browser handles this automatically. Django uses two cookies you'll see constantly:

| Cookie | Purpose |
|--------|---------|
| `sessionid` | Links the request to a server-side session (who is logged in) |
| `csrftoken` | CSRF protection token, verified on POST requests |

In Django: `request.COOKIES` gives you all cookies as a dictionary.

---

### Request Body Headers

**`Content-Type`** — The format of the request body.

```
Content-Type: application/x-www-form-urlencoded
Content-Type: application/json
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary7MA4YWxk
```

| Content-Type | Used for |
|-------------|----------|
| `application/x-www-form-urlencoded` | HTML form submissions |
| `application/json` | API requests with JSON body |
| `multipart/form-data` | File uploads |
| `text/plain` | Raw text |

In Django, the way you access the body depends on Content-Type:

```python
# application/x-www-form-urlencoded (HTML form)
title = request.POST.get('title')

# application/json (API client)
import json
data = json.loads(request.body)
title = data.get('title')

# multipart/form-data (file upload)
title = request.POST.get('title')
file  = request.FILES.get('image')
```

---

**`Content-Length`** — Size of the body in bytes.

```
Content-Length: 347
```

Tells the server how many bytes to read for the body. Required when a body is present.

---

### Security Headers

**`Referer`** *(yes, historically misspelled)* — The URL of the page that initiated this request.

```
Referer: https://www.example.com/articles/
```

When you click a link on `example.com/articles/`, the browser includes this header on the next request. Useful for analytics and — in Django — for validating CSRF tokens (Django checks that form submissions come from your own domain).

---

**`X-CSRFToken`** — Django's CSRF token sent via header (for AJAX requests).

```
X-CSRFToken: defgh456ijklmn789
```

HTML forms include the CSRF token in the body as a hidden field. JavaScript AJAX requests include it in this header instead. Django's CSRF middleware accepts both.

---

**`Origin`** — The origin (scheme + domain + port) of the request, used in CORS.

```
Origin: https://frontend.example.com
```

Sent by browsers for cross-origin requests. The server's CORS headers use this to decide whether to allow the request.

---

## The Request Body

The body carries data from client to server. It is only present when the method warrants it — POST, PUT, PATCH. Never on GET, HEAD, DELETE, or OPTIONS.

### Form Data

When an HTML form submits, the browser encodes field values as:

```
title=Hello+World&body=My+first+post&author=noting
```

Spaces become `+` or `%20`. Special characters are percent-encoded. This format is called **URL encoding** (`application/x-www-form-urlencoded`).

In Django, `request.POST` is a dictionary-like object containing all these values:
```python
request.POST['title']           # 'Hello World'
request.POST.get('title', '')   # Safe access with default
request.POST.getlist('tags')    # For multi-select fields
```

---

### JSON Body

Modern API clients send data as JSON:

```json
{
    "title": "Hello World",
    "body": "My first post",
    "tags": ["django", "python"],
    "published": true
}
```

Django doesn't parse JSON bodies automatically in standard views. You parse it yourself:

```python
import json

def create_article(request):
    if request.method == 'POST':
        try:
            data = json.loads(request.body)
        except json.JSONDecodeError:
            return JsonResponse({'error': 'Invalid JSON'}, status=400)

        title = data.get('title')
        ...
```

Django REST Framework handles JSON parsing automatically via serializers — one of its major conveniences.

---

### Multipart / File Upload

When a form includes a file input, the browser uses `multipart/form-data`:

```
------WebKitFormBoundary7MA4YWxk
Content-Disposition: form-data; name="title"

Hello World
------WebKitFormBoundary7MA4YWxk
Content-Disposition: form-data; name="avatar"; filename="photo.jpg"
Content-Type: image/jpeg

[binary image data]
------WebKitFormBoundary7MA4YWxk--
```

Each field is separated by a boundary string. Files carry their filename and MIME type.

In Django:
```python
# Form fields in request.POST, files in request.FILES
title  = request.POST.get('title')
avatar = request.FILES.get('avatar')

# request.FILES['avatar'] is an InMemoryUploadedFile or TemporaryUploadedFile
avatar.name          # 'photo.jpg'
avatar.content_type  # 'image/jpeg'
avatar.size          # size in bytes
```

The HTML form must have `enctype="multipart/form-data"` or file data will not be sent correctly.

## Django's Request Object

When Django receives an HTTP request, it parses the entire thing and wraps it in a **`HttpRequest`** object. This object is what your view function receives as the `request` parameter.

Every piece of the HTTP request is accessible through this object:

```python
def my_view(request):
    # Request line
    request.method          # 'GET', 'POST', etc.
    request.path            # '/articles/42/'
    request.get_full_path() # '/articles/42/?page=2'

    # Query string
    request.GET             # QueryDict: {'page': ['2'], 'tag': ['django']}

    # Body (POST form data)
    request.POST            # QueryDict: {'title': ['Hello'], ...}

    # Body (raw bytes — for JSON)
    request.body            # b'{"title": "Hello"}'

    # Files
    request.FILES           # MultiValueDict of uploaded files

    # Headers
    request.headers         # Case-insensitive dict: request.headers['Content-Type']
    request.META            # Full WSGI environment including HTTP_* headers

    # Authentication (set by middleware)
    request.user            # The logged-in User object (or AnonymousUser)
    request.session         # The session dict for this user

    # Network info
    request.META['REMOTE_ADDR']     # Client's IP address
    request.META['SERVER_NAME']     # Server hostname
    request.get_host()              # Domain from Host header
    request.scheme                  # 'http' or 'https'
    request.is_secure()             # True if HTTPS
```

This object is your window into the entire incoming request. Everything you need to build a response is here.

## Seeing Real Requests

The best way to internalize this is to read real requests.

**In Chrome DevTools:**
1. Open any site → F12 → Network tab
2. Click a request → Headers tab
3. Scroll to "Request Headers" — every header your browser sent
4. For POST forms, look at "Payload" or "Request Body"

**With curl (in a terminal):**
```bash
# See the request headers curl sends
curl -v https://httpbin.org/get

# Send a POST with JSON
curl -v -X POST https://httpbin.org/post \
     -H "Content-Type: application/json" \
     -d '{"title": "test"}'
```

`httpbin.org` is a free service that echoes back exactly what it receives — perfect for learning.

**With Django's debug toolbar:**
Once you install `django-debug-toolbar` in your project, every request shows full header and query information in an overlay panel.

## Key Terms

| Term | Definition |
|------|------------|
| **Request Line** | First line of an HTTP request: method, path, version |
| **Query String** | URL parameters after `?`, accessed via `request.GET` in Django |
| **Request Headers** | Metadata key-value pairs sent with every request |
| **Request Body** | The data payload of POST/PUT/PATCH requests |
| **Content-Type** | Header declaring the format of the request body |
| **URL Encoding** | Encoding format for HTML form data (`application/x-www-form-urlencoded`) |
| **Multipart** | Encoding format for file uploads (`multipart/form-data`) |
| **Cookie** | Browser-stored data sent automatically with every request |
| **Authorization** | Header carrying authentication credentials |
| **HttpRequest** | Django's Python object wrapping the entire incoming HTTP request |
| `request.GET` | QueryDict of URL query string parameters |
| `request.POST` | QueryDict of form body parameters |
| `request.FILES` | Dict of uploaded files |
| `request.body` | Raw bytes of the request body |
| `request.headers` | Case-insensitive dict of request headers |

---

## Summary

- An HTTP request consists of: **request line**, **headers**, a blank separator line, and an optional **body**
- The **request line** contains the method, path (including query string), and HTTP version
- **Headers** carry metadata: identity, content negotiation, authentication, cookies, and body format
- The **body** is present on POST, PUT, PATCH — formatted as URL-encoded, JSON, or multipart depending on `Content-Type`
- Django wraps the entire request in an `HttpRequest` object with clean Python access to every part
- Query parameters → `request.GET`, form body → `request.POST`, files → `request.FILES`, raw body → `request.body`
