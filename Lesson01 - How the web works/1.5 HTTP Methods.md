# 1.5 HTTP Methods

> Methods are the verbs of HTTP. They tell the server not just *what* you want, but *what you intend to do with it*.

---

## Why Methods Exist

HTTP could have been designed with a single "do something" request type. It wasn't — and for good reason.

When a client makes a request, the server needs to know the **intent** behind it. Fetching an article and deleting an article both target the same URL (`/articles/42/`). The method is what distinguishes them.

Methods give HTTP **semantic clarity** — a shared vocabulary of intent between clients and servers. This is what allows browsers, caches, proxies, load balancers, and your application code to all behave correctly without ambiguity.

---

## The Methods

### GET — Retrieve

**Purpose:** Fetch a resource without modifying anything on the server.

```
GET /articles/42/ HTTP/1.1
Host: www.example.com
```

Rules:
- No request body
- Must not change server state (no side effects)
- Safe and idempotent
- Responses can be cached

GET is the default. Typing a URL in a browser, clicking a link, loading an image — all GET requests. It is the most common HTTP method by volume on the web.

**In Django:**
```python
def article_detail(request, pk):
    if request.method == 'GET':
        article = get_object_or_404(Article, pk=pk)
        return render(request, 'articles/detail.html', {'article': article})
```

---

### POST — Submit / Create

**Purpose:** Send data to the server to create a resource or trigger an action.

```
POST /articles/ HTTP/1.1
Host: www.example.com
Content-Type: application/x-www-form-urlencoded

title=Django+for+Beginners&body=Let%27s+start+here
```

Rules:
- Has a request body containing the data being submitted
- May change server state
- Not idempotent — two identical POST requests typically create two resources
- Responses are generally not cached

POST is used for: form submissions, creating records, login, file uploads, triggering actions.

**In Django:**
```python
def create_article(request):
    if request.method == 'POST':
        form = ArticleForm(request.POST)
        if form.is_valid():
            form.save()
            return redirect('article_list')
    else:
        form = ArticleForm()
    return render(request, 'articles/create.html', {'form': form})
```

---

### PUT — Replace

**Purpose:** Replace an existing resource entirely with the provided data.

```
PUT /articles/42/ HTTP/1.1
Host: www.example.com
Content-Type: application/json

{"title": "New Title", "body": "New body", "author": 1}
```

Rules:
- Has a request body containing the complete new representation
- Idempotent — sending the same PUT twice results in the same state
- The entire resource is replaced, not partially updated

PUT means: *"Whatever is at this URL, replace it completely with what I'm sending."* If you send a PUT with only a `title` field and the resource has ten fields, the other nine fields are gone.

---

### PATCH — Update Partially

**Purpose:** Apply a partial update to an existing resource.

```
PATCH /articles/42/ HTTP/1.1
Host: www.example.com
Content-Type: application/json

{"title": "Updated Title Only"}
```

Rules:
- Has a request body containing only the fields to change
- Not necessarily idempotent (depends on implementation)
- Only the provided fields are modified; everything else remains unchanged

PATCH means: *"Update just the parts I'm sending. Leave the rest alone."*

**PUT vs PATCH — the practical distinction:**

| Scenario | Use |
|----------|-----|
| Replacing an entire document | PUT |
| Updating a user's email address | PATCH |
| Editing one field of a form with 20 fields | PATCH |
| Uploading a new version of a file | PUT |

In most real-world Django REST APIs, PATCH is more commonly used than PUT because partial updates are more practical and less error-prone.

---

### DELETE — Remove

**Purpose:** Delete the specified resource.

```
DELETE /articles/42/ HTTP/1.1
Host: www.example.com
```

Rules:
- Typically no request body
- Idempotent — deleting something that's already deleted still results in it not existing
- Typically returns `204 No Content` (success, nothing to return) or `200 OK`

**In Django REST Framework:**
```python
class ArticleDetail(APIView):
    def delete(self, request, pk):
        article = get_object_or_404(Article, pk=pk)
        article.delete()
        return Response(status=status.HTTP_204_NO_CONTENT)
```

---

### HEAD — Headers Only

**Purpose:** Identical to GET, but the server returns only headers — no body.

```
HEAD /articles/42/ HTTP/1.1
Host: www.example.com
```

Use cases:
- Check if a resource exists without downloading it
- Check when a resource was last modified (`Last-Modified` header)
- Check the size of a file before downloading (`Content-Length` header)

If your Django view correctly handles GET, HEAD is automatically supported — Django handles it.

---

### OPTIONS — Capabilities

**Purpose:** Ask the server what methods are allowed for a given URL.

```
OPTIONS /articles/42/ HTTP/1.1
Host: www.example.com
```

Response:
```
HTTP/1.1 200 OK
Allow: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS
```

OPTIONS is primarily used in two contexts:
1. **CORS preflight requests** — browsers send an OPTIONS request before cross-origin requests to verify the server allows them
2. **API discovery** — clients checking what operations are available

In Django REST Framework, OPTIONS requests are handled automatically and return detailed API schema information.

## Safety and Idempotency

Two properties that define how methods behave — and how clients, caches, and proxies are allowed to treat them.

### Safe Methods

A **safe** method does not modify server state. It is a read-only operation.

Safe methods: **GET, HEAD, OPTIONS**

Because they're safe, caches are allowed to store their responses. Browsers can re-send them freely. A prefetch or speculative fetch won't cause harm.

### Idempotent Methods

An **idempotent** method produces the same server state regardless of how many times it is called.

Idempotent methods: **GET, HEAD, PUT, DELETE, OPTIONS**

```
DELETE /articles/42/  →  Article 42 deleted (200)
DELETE /articles/42/  →  Article 42 already gone (404)
```

Both calls result in the same state: article 42 does not exist. The response codes differ, but the state is identical. That's idempotency.

POST and PATCH are **not** idempotent:
- `POST /articles/` called three times → three articles created
- `PATCH /articles/42/` with `{"views": views + 1}` called three times → views incremented three times

| Method | Safe | Idempotent |
|--------|------|-----------|
| GET | ✓ | ✓ |
| HEAD | ✓ | ✓ |
| OPTIONS | ✓ | ✓ |
| PUT | ✗ | ✓ |
| DELETE | ✗ | ✓ |
| POST | ✗ | ✗ |
| PATCH | ✗ | ✗ |

This table matters in practice. When a network request fails and you need to decide whether to retry automatically, idempotency is what tells you it's safe to do so.

## The Browser's Limitation

Standard HTML gives browsers only two methods: **GET** and **POST**.

- `<a href="...">` → GET
- `<form method="get">` → GET
- `<form method="post">` → POST
- There is no `<form method="delete">` or `<form method="put">` in HTML

This is a historical limitation of the web's original design.

### How Django Works Around It

For traditional Django views (not APIs), Django uses a convention called **method spoofing** via a hidden form field:

```html
<form method="post" action="/articles/42/delete/">
    {% csrf_token %}
    <button type="submit">Delete</button>
</form>
```

The form submits a POST, but the URL and view logic make the intent clear. Django doesn't need true DELETE requests for form-based views — POST + meaningful URL is sufficient.

For proper REST APIs with PUT, PATCH, and DELETE, you use JavaScript's `fetch()` API or DRF's `APIView`, which accepts all methods:

```javascript
fetch('/api/articles/42/', {
    method: 'DELETE',
    headers: {
        'X-CSRFToken': getCookie('csrftoken')
    }
})
```

## Methods in Django URL Design

A well-designed Django URL structure uses a single URL with multiple methods for resource operations:

```
/articles/           GET    → list all articles
                     POST   → create a new article

/articles/42/        GET    → retrieve article 42
                     PUT    → replace article 42
                     PATCH  → partially update article 42
                     DELETE → delete article 42
```

This pattern — **one URL, multiple methods** — is the foundation of **REST** (Representational State Transfer), the architectural style behind most modern APIs.

In pure Django views:
```python
# urls.py
path('articles/<int:pk>/', views.article_detail),

# views.py
def article_detail(request, pk):
    article = get_object_or_404(Article, pk=pk)

    if request.method == 'GET':
        return render(request, 'detail.html', {'article': article})

    elif request.method == 'POST':
        # handle update (form-based)
        ...

    elif request.method == 'DELETE':
        # handle delete (API / JS request)
        article.delete()
        return HttpResponse(status=204)
```

In Django REST Framework:
```python
class ArticleDetail(APIView):
    def get(self, request, pk):    ...
    def put(self, request, pk):    ...
    def patch(self, request, pk):  ...
    def delete(self, request, pk): ...
```

## Key Terms

| Term | Definition |
|------|------------|
| **HTTP Method** | The verb in a request indicating what action to perform |
| **GET** | Retrieve a resource without side effects |
| **POST** | Submit data to create a resource or trigger an action |
| **PUT** | Replace a resource entirely |
| **PATCH** | Partially update a resource |
| **DELETE** | Remove a resource |
| **HEAD** | Like GET but returns headers only |
| **OPTIONS** | Query what methods a URL supports |
| **Safe** | A method that does not modify server state |
| **Idempotent** | A method that produces the same state no matter how many times it's called |
| **CORS** | Cross-Origin Resource Sharing — a browser security mechanism involving OPTIONS preflight |

---

## Summary

- HTTP methods define the **intent** of a request, not just the target
- **GET** retrieves, **POST** creates/submits, **PUT** replaces, **PATCH** partially updates, **DELETE** removes
- **Safe** methods (GET, HEAD, OPTIONS) don't modify state; **idempotent** methods (GET, HEAD, PUT, DELETE, OPTIONS) produce the same state no matter how many times they're called
- HTML forms support only GET and POST — everything else requires JavaScript or DRF
- REST APIs use a **single URL with multiple methods** for clean resource design
- Django accesses the method via `request.method`; DRF routes by defining `get()`, `post()`, etc. on a view class
