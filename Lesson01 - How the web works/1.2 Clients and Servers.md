# 1.2 Clients and Servers

> Two roles. One relationship. Everything on the web is built on this.

---

## The Fundamental Model

The web is built on a single, simple idea: **one machine asks, another machine answers**.

The machine that asks is called the **client**.
The machine that answers is called the **server**.

This is called the **client-server model**, and it is the architectural foundation of every web application you will ever build.

---

## What is a Client?

A **client** is any program that initiates a request to a server.

The most familiar client is a **web browser** — Chrome, Firefox, Safari. But a client can be many things:

- A mobile app requesting data from an API
- A Python script using the `requests` library to fetch a webpage
- `curl` running in a terminal
- One server making a request to another server
- Django's test client (used in automated tests)

What makes something a client is **behaviour**, not form: it sends a request and waits for a response.

---

## What is a Server?

A **server** is a program that listens for incoming requests and responds to them.

Common examples:

| Server | Role |
|--------|------|
| **Nginx** | Web server — serves static files, proxies requests |
| **Gunicorn** | WSGI server — runs Python web applications |
| **Django** | Application server — processes requests, returns responses |
| **PostgreSQL** | Database server — responds to queries |
| **Redis** | Cache server — returns stored values |

Notice: "server" refers to **software**, not necessarily hardware. Multiple server programs often run on the same physical machine.

> A single computer can be both a client and a server at the same time — for example, a Django application (acting as a server to the browser) might itself be a client to a PostgreSQL database server.

---

## The Request-Response Cycle

The communication between client and server always follows the same pattern:

```
        CLIENT                          SERVER
           │                               │
           │   1. Client sends request     │
           │ ─────────────────────────────►│
           │                               │  2. Server processes
           │                               │     the request
           │   3. Server sends response    │
           │ ◄─────────────────────────────│
           │                               │
           │  4. Client handles response   │
           │                               │
```

This is the **request-response cycle**. It is synchronous: the client waits while the server processes, and the server only responds when it has something to return.

Every page load you've ever seen, every form you've ever submitted, every API call ever made — all of them follow this cycle.

---

## Key Properties of the Client-Server Relationship

### 1. The client always initiates

Servers do not spontaneously send data to clients (in the standard HTTP model). A client must always make the first move. This is why, for example, a chat application needs tricks like **WebSockets** or **long-polling** to push new messages to the browser — HTTP by itself doesn't support server-initiated communication.

### 2. The server is always listening

A server is a program in a permanent listen state, waiting on a **port** for incoming connections. Your browser connects to port 80 (HTTP) or port 443 (HTTPS). A database server listens on port 5432 (PostgreSQL default). Redis listens on 6379.

### 3. One server, many clients

A single server handles requests from potentially millions of clients concurrently. This is why performance, scalability, and efficient code matter in backend development.

### 4. They don't need to know each other in advance

Any browser anywhere in the world can make a request to your Django server. The client and server don't have a persistent relationship — they communicate only for the duration of a request-response cycle, then go their separate ways.

---

## Statelessness

HTTP is **stateless**. This means the server treats each request as completely independent. It has no built-in memory of previous requests.

```
Request 1:  "Give me the homepage"      → Server responds
Request 2:  "Give me my profile page"   → Server has no idea who made request 1
```

From the server's perspective, these two requests could come from entirely different people, even if they come from the same browser seconds apart.

This has a profound implication: **if you want the server to remember something about a user (like the fact they're logged in), you have to implement that memory yourself**, using tools like sessions, cookies, or tokens.

Django has built-in systems for all of this — but understanding *why* they exist comes from understanding statelessness first.

---

## Where Django Fits

In a typical Django deployment, there are actually several layers:

```
Browser (Client)
       │
       ▼
  Nginx (Web Server)
  — handles static files
  — terminates HTTPS
  — forwards dynamic requests
       │
       ▼
  Gunicorn (WSGI Server)
  — runs the Python process
  — manages worker processes
       │
       ▼
  Django (Application)
  — matches URLs
  — runs view functions
  — queries the database
  — returns a response
       │
       ▼
  PostgreSQL (Database Server)
  — responds to SQL queries
```

Django sits in the middle of this chain. It is a **server** from the browser's perspective, but a **client** from the database's perspective.

When you're learning Django locally, you use Django's built-in development server (`python manage.py runserver`), which collapses several of these layers into one for simplicity. In production, they're separated for performance, security, and reliability.

---

## A Concrete Example

You visit `https://yourapp.com/dashboard/`:

1. **Browser** (client) sends an HTTP GET request to `yourapp.com`
2. **Nginx** receives it, sees the path `/dashboard/`, forwards it to Gunicorn
3. **Gunicorn** passes it to Django
4. **Django** matches the URL `/dashboard/` to a view function
5. The view function (your code) queries **PostgreSQL** — Django is now the client, PostgreSQL is the server
6. PostgreSQL returns the user's data
7. Django renders an HTML template with the data
8. Django returns an HTTP response
9. The response travels back through Gunicorn → Nginx → internet → browser
10. The browser renders the HTML

You wrote the code in step 4–7. Everything else is infrastructure.

## Key Terms

| Term | Definition |
|------|------------|
| **Client** | A program that sends requests to a server |
| **Server** | A program that listens for and responds to requests |
| **Client-Server Model** | An architecture where one party requests and another responds |
| **Request-Response Cycle** | The pattern of client sends → server processes → server replies |
| **Stateless** | Each request is independent; the server has no memory of prior requests |
| **Port** | A numbered channel on a machine where a specific service listens |
| **WSGI** | Web Server Gateway Interface — the Python standard for connecting web servers to Python apps |

---

## Summary

- The web is built on the **client-server model**: clients ask, servers answer
- A **client** initiates requests; a **server** listens and responds
- The **request-response cycle** is the fundamental unit of web communication
- HTTP is **stateless** — the server remembers nothing between requests by default
- In a Django app, Django is a server to the browser and a client to the database
- Django's development server simplifies this for local development; production uses multiple layers
