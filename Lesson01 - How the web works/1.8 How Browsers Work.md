# 1.8 How Browsers Work

> The browser is your application's final executor. Understanding what it does with your response makes you write better templates, better static files, and better Django views.

---

## The Browser is Not Just a Window

Most developers think of the browser as a display — something that shows the page your server returned. That's a fraction of what it does.

The browser is a sophisticated runtime engine that:
- Parses and validates HTML into a tree structure
- Fetches dozens of additional resources in parallel
- Applies CSS to compute the exact visual appearance of every element
- Executes JavaScript that may rewrite the entire page
- Manages secure storage (cookies, localStorage, IndexedDB)
- Enforces security policies (CORS, CSP, HTTPS requirements)
- Caches responses according to server instructions
- Handles forms, navigation, and history

Understanding this is essential because your Django views produce the inputs to this process. Every template you write, every static file you serve, every header you set — the browser is the consumer of all of it.

## The Rendering Pipeline

When your Django view returns an HTML response, the browser executes a precise sequence of steps to turn that HTML into pixels on screen.

### Step 1 — Receive the HTML

The browser begins parsing HTML as it streams in — it doesn't wait for the entire response before starting. This is why `Content-Length` and streaming matter, and why placing `<script>` tags at the end of `<body>` (or using `defer`) is a performance best practice.

---

### Step 2 — Parse HTML → DOM

The browser's HTML parser reads the markup and builds the **DOM (Document Object Model)** — a tree of objects representing every element in the document.

```html
<!doctype html>
<html>
  <head>
    <title>Articles</title>
  </head>
  <body>
    <h1>Latest Articles</h1>
    <ul>
      <li><a href="/articles/1/">Django Intro</a></li>
    </ul>
  </body>
</html>
```

Becomes this DOM tree:

```
Document
└── html
    ├── head
    │   └── title
    │       └── "Articles"
    └── body
        ├── h1
        │   └── "Latest Articles"
        └── ul
            └── li
                └── a [href="/articles/1/"]
                    └── "Django Intro"
```

JavaScript interacts with the page by reading and modifying this tree. `document.querySelector('h1')` navigates to the `h1` node. `element.textContent = 'New Title'` modifies it.

---

### Step 3 — Discover and Fetch Sub-Resources

As the parser reads the HTML, it finds references to other resources and starts fetching them immediately:

```html
<link rel="stylesheet" href="/static/css/main.css">   ← fetch CSS
<script src="/static/js/app.js"></script>              ← fetch JS
<img src="/media/photo.jpg" alt="...">                 ← fetch image
```

Each of these triggers a new HTTP request — with its own DNS lookup (if a different domain), TCP connection, and request-response cycle. A typical page load involves 20–100+ individual HTTP requests.

This is why Django's static file system matters. Django serves static files correctly in development via `django.contrib.staticfiles`. In production, Nginx serves them directly — much faster — while Django handles only dynamic requests.

---

### Step 4 — Parse CSS → CSSOM

While the HTML is parsing, fetched CSS files are parsed into the **CSSOM (CSS Object Model)** — a tree representing all style rules and how they apply to elements.

```css
h1 { font-size: 2rem; color: #111; }
ul li a { color: #0066cc; text-decoration: none; }
```

The browser computes **computed styles** for every element — the final values after cascading, inheritance, specificity, and media queries are all resolved.

**CSS blocks rendering.** The browser won't paint content until all CSS is downloaded and parsed. This is why CSS files belong in `<head>` — so they're fetched early and don't block the initial render.

---

### Step 5 — JavaScript Parsing and Execution

JavaScript files are parsed and executed by the browser's JavaScript engine (V8 in Chrome, SpiderMonkey in Firefox).

**Script loading behaviour:**

```html
<!-- Blocks HTML parsing while downloading and executing -->
<script src="app.js"></script>

<!-- Downloads in parallel, executes after HTML is parsed -->
<script src="app.js" defer></script>

<!-- Downloads in parallel, executes immediately when ready -->
<script src="app.js" async></script>
```

A blocking script in `<head>` without `defer` or `async` halts HTML parsing until the script downloads and executes. This is why `defer` is the modern best practice for most scripts, and why older advice says to put scripts at the end of `<body>`.

**JavaScript can modify the DOM.** Once executed, JS can add, remove, or change any element. Modern single-page applications (SPAs) take this to an extreme — the server returns a near-empty HTML shell, and JavaScript builds the entire UI.

---

### Step 6 — Render Tree Construction

With both the DOM and CSSOM built, the browser combines them into a **Render Tree** — the DOM with all its computed styles, minus invisible elements (`display: none`, `<head>` contents, etc.).

```
Render Tree (visible elements only, with styles):
└── body [background: white; font-family: sans-serif]
    ├── h1 [font-size: 2rem; color: #111]
    │   └── "Latest Articles"
    └── ul
        └── li
            └── a [color: #0066cc]
                └── "Django Intro"
```

---

### Step 7 — Layout (Reflow)

The browser calculates the **exact position and size** of every element in the render tree — in pixels. This accounts for the viewport size, box model, flexbox/grid rules, float, and all other layout properties.

This step is called **layout** or **reflow**. It is computationally expensive, which is why causing many layout recalculations via JavaScript (e.g., reading `element.offsetWidth` inside a loop that also modifies styles) tanks performance.

---

### Step 8 — Paint

The browser converts layout information into actual pixel colors — rasterizing text, borders, backgrounds, shadows, images.

The result is a series of layers that are composited (combined) into the final image you see. GPU-accelerated properties like `transform` and `opacity` skip the layout and paint steps — they only affect compositing, making them the fastest properties to animate.

---

### Step 9 — Composite and Display

The painted layers are sent to the GPU, composited into a final frame, and displayed on screen.

For a smooth 60fps experience, this entire pipeline — for every frame — must complete in under **16 milliseconds**.

---

## The Complete Pipeline

```
HTML response received
        │
        ▼
Parse HTML → DOM
        │
        ├──► Discover resources (CSS, JS, images) → HTTP requests
        │
        ▼
Parse CSS → CSSOM
        │
        ▼
Execute JavaScript (may modify DOM/CSSOM)
        │
        ▼
DOM + CSSOM → Render Tree
        │
        ▼
Layout (positions and sizes)
        │
        ▼
Paint (pixels)
        │
        ▼
Composite → Screen
```

## The Network Tab — Your Most Important Tool

The browser's developer tools expose everything happening under the hood. The **Network tab** is where backend developers spend most of their debugging time.

Open it: `F12` → Network tab → reload the page.

You'll see:

```
Name                    Status   Type        Size    Time
─────────────────────────────────────────────────────────
/                       200      document    14 KB   120ms
/static/css/main.css    200      stylesheet  8 KB    45ms
/static/js/app.js       200      script      32 KB   89ms
/media/hero.jpg         200      jpeg        280 KB  340ms
/api/articles/          200      json        2 KB    67ms
/favicon.ico            404      text/html   1 KB    12ms
```

Each row is one HTTP request. Click any row to see:

- **Headers** tab: every request header your browser sent, every response header the server returned, the status code, the method
- **Response** tab: the full response body
- **Timing** tab: breakdown of time spent on DNS, TCP connection, TLS handshake, waiting (TTFB), and download

**What to look for as a Django developer:**

| What you see | What it means |
|-------------|---------------|
| 404 on a static file | `STATIC_URL` misconfigured, or `collectstatic` not run |
| 500 on any request | Unhandled exception in your view — check Django's error page |
| 302 redirect loop | Redirect condition is always true |
| Long TTFB (>500ms) | Slow database query, missing index, or N+1 query problem |
| 403 on API call | CSRF token missing, or permission denied |
| Missing `Content-Type` | Response not setting content type explicitly |

## How the Browser Handles Forms

When a user submits an HTML form, the browser:

1. Collects all input values
2. Serialises them based on the form's `enctype`
3. Sends an HTTP request with method from the `method` attribute

```html
<form method="post" action="/articles/create/" enctype="multipart/form-data">
    {% csrf_token %}
    <input type="text" name="title">
    <input type="file" name="image">
    <button type="submit">Create</button>
</form>
```

- `method="post"` → HTTP POST
- `action="/articles/create/"` → URL of the request
- `enctype="multipart/form-data"` → required when the form contains file inputs

Without `enctype="multipart/form-data"`, file inputs are not included in the request body. Django's `request.FILES` will be empty. This is one of the most common beginner mistakes with file uploads.

The `{% csrf_token %}` template tag inserts a hidden input field with the CSRF token. The browser sends it with the form data. Django's `CsrfViewMiddleware` verifies it on the server.

## How the Browser Handles Cookies

Cookies are the mechanism that gives HTTP its "memory" — they persist state across stateless requests.

**How cookies are set:**

The server includes `Set-Cookie` in a response header:
```
Set-Cookie: sessionid=abc123; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=1209600
```

The browser stores this cookie and automatically includes it in all future requests to the same domain:
```
Cookie: sessionid=abc123
```

**Key browser rules for cookies:**

- Cookies are scoped by **domain** and **path** — `example.com` cookies are not sent to `other.com`
- `HttpOnly` cookies are invisible to JavaScript — `document.cookie` won't show them
- `Secure` cookies are only sent over HTTPS — never over HTTP
- `SameSite=Lax` cookies are not sent on cross-site POST requests — CSRF protection
- Session cookies (no `Max-Age` or `Expires`) are deleted when the browser closes
- Persistent cookies (with `Max-Age`) survive browser restarts until they expire

Django's `sessionid` cookie is what keeps you logged in. When the middleware reads it, it looks up the corresponding session data on the server and populates `request.user`.

---

## Browser Storage Beyond Cookies

The browser provides several storage mechanisms beyond cookies, each with different characteristics:

| Storage | Capacity | Accessible to JS | Sent with requests | Persists |
|---------|----------|-----------------|---------------------|---------|
| **Cookie** | ~4KB | Yes (unless HttpOnly) | Yes (automatically) | Configurable |
| **localStorage** | ~5MB | Yes | No | Until cleared |
| **sessionStorage** | ~5MB | Yes | No | Until tab closes |
| **IndexedDB** | Large | Yes | No | Until cleared |

Django's session system uses **cookies** to store the session ID — which the browser sends with every request — and stores the actual session data server-side. This is the correct approach for authentication: the server is the source of truth.

`localStorage` and `sessionStorage` are JavaScript-accessible, not sent with requests, and can't be made HttpOnly. They're appropriate for UI preferences and caching non-sensitive data client-side — not for authentication tokens in a security-sensitive application.

## CORS — Cross-Origin Resource Sharing

When JavaScript in the browser tries to make an HTTP request to a **different domain** than the page it's on, the browser enforces **CORS (Cross-Origin Resource Sharing)** rules.

```
Page loaded from:   https://frontend.example.com
JS fetch to:        https://api.example.com/articles/
                              ↑
                    Different origin — CORS applies
```

An **origin** is: scheme + domain + port. `https://example.com` and `http://example.com` are different origins. `https://example.com` and `https://api.example.com` are different origins.

**How it works:**

For "simple" requests, the browser sends the request and checks the response:

```
Response must include:
Access-Control-Allow-Origin: https://frontend.example.com
```

For "preflighted" requests (non-GET/POST, or custom headers), the browser first sends an OPTIONS request:

```
OPTIONS /api/articles/ HTTP/1.1
Origin: https://frontend.example.com
Access-Control-Request-Method: DELETE
```

The server must respond with permission before the real request proceeds.

In Django, CORS is handled by the `django-cors-headers` package:

```python
# settings.py
INSTALLED_APPS = [..., 'corsheaders']
MIDDLEWARE = ['corsheaders.middleware.CorsMiddleware', ...]

CORS_ALLOWED_ORIGINS = [
    "https://frontend.example.com",
]
```

CORS errors appear in the browser console as:
```
Access to fetch at 'https://api.example.com/articles/' from origin
'https://frontend.example.com' has been blocked by CORS policy.
```

This is one of the most common errors when building a Django API consumed by a JavaScript frontend.

## Key Terms

| Term | Definition |
|------|------------|
| **DOM** | Document Object Model — the browser's tree representation of parsed HTML |
| **CSSOM** | CSS Object Model — the browser's tree representation of parsed CSS |
| **Render Tree** | The DOM with computed styles applied, minus invisible elements |
| **Layout / Reflow** | Computing the exact position and size of every element |
| **Paint** | Converting layout to pixels |
| **TTFB** | Time To First Byte — time between request sent and first byte of response received |
| **defer** | Script attribute: download in parallel, execute after HTML is parsed |
| **async** | Script attribute: download in parallel, execute immediately when ready |
| **CORS** | Cross-Origin Resource Sharing — browser security policy for cross-domain requests |
| **Origin** | Scheme + domain + port combination |
| **Cookie** | Small piece of data stored by the browser and sent with every request |
| **HttpOnly** | Cookie attribute preventing JavaScript access |
| **SameSite** | Cookie attribute controlling cross-site sending behaviour |
| **Network Tab** | Browser DevTools panel showing all HTTP requests made by the page |

## Summary

- The browser is a full runtime: it parses HTML, fetches resources, applies CSS, executes JavaScript, and composites the final image
- The rendering pipeline: HTML → DOM → fetch sub-resources → CSS → CSSOM → JavaScript → Render Tree → Layout → Paint → Screen
- A page load triggers many HTTP requests — each CSS, JS, and image file is a separate request
- The **Network tab** in DevTools lets you see every request, its headers, status code, and timing — essential for debugging Django apps
- Browsers use **cookies** to carry session state with every request; Django's session middleware reads the `sessionid` cookie to identify users
- **CORS** is a browser security policy enforced on cross-origin JavaScript requests; `django-cors-headers` configures Django's CORS response headers
- Understanding the browser's behaviour explains why Django's template, static file, CSRF, and security header systems are designed the way they are
