# 1.9 What Happens on the Server Side

> The browser's job ends when it sends the request. Everything that happens next — until the response arrives — is the server's world. This is where Django lives.

---

## The Server's Job

From the browser's perspective, a server is a black box: you send it a request, it sends back a response. What happens inside is irrelevant to the browser.

From your perspective as a Django developer, the inside is everything.

When a request arrives at a server, a chain of software components handles it — each with a distinct responsibility. Understanding this chain explains:

- Why Django can't be deployed the same way in production as in development
- What Nginx, Gunicorn, and WSGI are and why they exist
- What Django actually does and what it doesn't
- Why `python manage.py runserver` is not for production

---

## Static vs Dynamic Responses

The first distinction every server makes: is this request for a **static** or **dynamic** resource?

### Static Resources

A **static resource** is a file that exists on disk exactly as it will be sent. It doesn't change based on who's asking or when.

Examples:
- CSS files: `/static/css/main.css`
- JavaScript files: `/static/js/app.js`
- Images: `/media/uploads/photo.jpg`
- HTML files in simple sites: `/about.html`

Serving a static file is trivial: find the file on disk, read it, send it. No computation, no database, no Python.

### Dynamic Resources

A **dynamic resource** is a response that is **built on request** — its content depends on factors like who is logged in, what's in the database, what parameters were passed.

Examples:
- A user's profile page (content depends on which user)
- A search results page (content depends on the query)
- An article list (content depends on what's in the database)
- An API endpoint (content depends on the query and the authenticated user)

This is where Django lives. Every view you write is producing a dynamic response.

## The Production Server Stack

In production, these two resource types are handled by different software:

```
Internet
    │
    ▼
┌─────────────────────────────────────────┐
│               NGINX                     │
│  (Web Server / Reverse Proxy)           │
│                                         │
│  /static/*  →  serve file from disk     │
│  /media/*   →  serve file from disk     │
│  everything else → forward to Gunicorn  │
└──────────────────┬──────────────────────┘
                   │ (dynamic requests only)
                   ▼
┌─────────────────────────────────────────┐
│              GUNICORN                   │
│  (WSGI Server / Process Manager)        │
│                                         │
│  Manages multiple Python worker processes│
│  Passes each request to Django          │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│               DJANGO                    │
│  (Python Web Application)               │
│                                         │
│  Middleware → URL routing → View        │
│  View → Database → Template → Response  │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│            POSTGRESQL                   │
│  (Database Server)                      │
│                                         │
│  Responds to Django's SQL queries       │
└─────────────────────────────────────────┘
```

Each layer has a specific job. They communicate via well-defined interfaces.

## Nginx — The Web Server

**Nginx** (pronounced "engine-x") is a high-performance web server and reverse proxy. In a Django production setup, it sits at the front of the entire stack.

### What Nginx Does

**1. Terminates HTTPS**

Nginx handles the TLS handshake and encryption/decryption. Django receives plain HTTP from Gunicorn — it doesn't deal with TLS at all. This is simpler and more efficient.

**2. Serves static files directly**

Nginx reads files from disk and sends them without involving Python at all. Serving a static file in Nginx takes microseconds and handles thousands of concurrent requests. If Django served static files, each one would spin up a Python process — 100× slower and a major bottleneck.

```nginx
# Nginx config (simplified)
server {
    listen 443 ssl;
    server_name example.com;

    # Static files — served directly
    location /static/ {
        alias /var/www/myapp/staticfiles/;
    }

    location /media/ {
        alias /var/www/myapp/media/;
    }

    # Everything else — forward to Django via Gunicorn
    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

**3. Rate limiting and access control**

Nginx can block IPs, limit request rates, require authentication for certain paths — all before a request ever reaches Python.

**4. Load balancing**

Nginx can distribute requests across multiple Gunicorn processes or even multiple servers.

## WSGI — The Interface Between Web Servers and Python

**WSGI (Web Server Gateway Interface)** is a Python standard (PEP 3333) that defines how web servers communicate with Python web applications.

Before WSGI, every Python framework required a different server adapter. WSGI unified this: any WSGI-compatible server (Gunicorn, uWSGI, mod_wsgi) can run any WSGI-compatible application (Django, Flask, FastAPI).

### What WSGI Defines

A WSGI application is a Python callable (a function or a class with `__call__`) that:

1. Accepts two arguments: `environ` (a dict of request data) and `start_response` (a callable to send the status line and headers)
2. Returns an iterable of byte strings (the response body)

Django's WSGI application entry point lives in `wsgi.py`:

```python
# myproject/wsgi.py
import os
from django.core.wsgi import get_wsgi_application

os.environ.setdefault('DJANGO_SETTINGS_MODULE', 'myproject.settings')

application = get_wsgi_application()
```

This `application` object is what Gunicorn calls for every request.

---

## Gunicorn — The WSGI Server

**Gunicorn** (Green Unicorn) is a Python WSGI HTTP server. It manages the process of running a Django application at scale.

### What Gunicorn Does

**1. Manages worker processes**

Gunicorn forks multiple Python worker processes — typically 2–4 per CPU core. Each worker handles one request at a time. While one worker is waiting for a database query, other workers handle other requests.

```
Gunicorn (master process)
    │
    ├── Worker 1  ← handling request from user A
    ├── Worker 2  ← handling request from user B
    ├── Worker 3  ← idle, ready
    └── Worker 4  ← idle, ready
```

**2. Translates HTTP to WSGI**

Gunicorn receives an HTTP request from Nginx, parses it into a WSGI `environ` dictionary, and passes it to Django.

**3. Handles worker lifecycle**

Gunicorn monitors workers and restarts them if they crash or exceed memory limits.

### Starting Gunicorn

```bash
gunicorn myproject.wsgi:application \
    --workers 4 \
    --bind 127.0.0.1:8000 \
    --timeout 30
```

### ASGI — The Modern Alternative

WSGI is synchronous — each worker handles one request at a time. For high-concurrency workloads (many simultaneous connections, WebSockets, long-polling), **ASGI (Asynchronous Server Gateway Interface)** is the modern replacement.

Django supports ASGI natively. The ASGI server equivalent of Gunicorn is **Uvicorn** or **Daphne**.

```python
# myproject/asgi.py
import os
from django.core.asgi import get_asgi_application

os.environ.setdefault('DJANGO_SETTINGS_MODULE', 'myproject.settings')

application = get_asgi_application()
```

For most Django projects (standard request-response, no WebSockets), WSGI/Gunicorn is sufficient and simpler. ASGI becomes important for real-time features.

## What Django Actually Does

When Gunicorn passes a WSGI request to Django, Django's internal machinery takes over.

### Phase 1 — Request Processing

```python
# Django receives the environ dict from Gunicorn
# and creates an HttpRequest object
request = WSGIRequest(environ)
```

Django wraps all the raw WSGI data into the `HttpRequest` object you work with in views.

### Phase 2 — Middleware (Request Phase)

Each middleware in `MIDDLEWARE` setting processes the request in order:

```python
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.common.CommonMiddleware',
    'django.middleware.csrf.CsrfViewMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware',
    'django.contrib.messages.middleware.MessageMiddleware',
    'django.middleware.clickjacking.XFrameOptionsMiddleware',
]
```

What each one does on the request:

| Middleware | What it does on request |
|-----------|------------------------|
| `SecurityMiddleware` | Redirects HTTP → HTTPS, sets HSTS headers |
| `SessionMiddleware` | Reads the `sessionid` cookie, loads the session data |
| `CommonMiddleware` | Normalises URLs (appends trailing slash if needed) |
| `CsrfViewMiddleware` | Validates CSRF token on POST requests |
| `AuthenticationMiddleware` | Reads the session, populates `request.user` |
| `MessageMiddleware` | Loads one-time flash messages from the session |
| `XFrameOptionsMiddleware` | (response only) |

Each middleware can short-circuit the chain and return a response early — CSRF failure returns a `403 Forbidden` before the view ever runs.

### Phase 3 — URL Resolution

Django's URL dispatcher (`URLResolver`) matches the request path against URL patterns in `urls.py`:

```python
# urls.py
urlpatterns = [
    path('articles/', views.article_list),
    path('articles/<int:pk>/', views.article_detail),
    path('articles/create/', views.create_article),
]
```

Django tries each pattern in order. The first match wins. If no pattern matches, Django raises a `Http404`.

URL patterns can capture values from the path:
```
/articles/42/  →  article_detail view, with pk=42
```

### Phase 4 — View Execution

Django calls the matched view function, passing the `HttpRequest` and any captured URL parameters:

```python
def article_detail(request, pk):
    # request = the HttpRequest object
    # pk = 42 (captured from the URL)

    article = Article.objects.get(pk=pk)
    return render(request, 'articles/detail.html', {'article': article})
```

Inside the view, Python code runs: database queries execute, business logic processes, a template renders or JSON serialises. The view **must** return an `HttpResponse`.

### Phase 5 — Middleware (Response Phase)

The response passes back through middleware in reverse order. Each middleware can modify the response:

| Middleware | What it does on response |
|-----------|-------------------------|
| `XFrameOptionsMiddleware` | Adds `X-Frame-Options: DENY` header |
| `SecurityMiddleware` | Adds `Strict-Transport-Security`, `X-Content-Type-Options` |
| `SessionMiddleware` | Saves modified session data, sets `sessionid` cookie if new |

### Phase 6 — Response Back to Gunicorn

Django returns the `HttpResponse` object. Gunicorn serialises it back into an HTTP response and sends it to Nginx, which sends it to the browser.

## Django's Development Server

`python manage.py runserver` starts Django's built-in development server.

It is **not** Gunicorn + Nginx. It is a single-threaded Python HTTP server that handles all roles itself:

```
Browser
    │
    ▼
Django's runserver
    │
    ├── Serves static files (via staticfiles app)
    ├── Parses HTTP
    ├── Creates HttpRequest
    ├── Runs middleware
    ├── Routes URLs
    ├── Calls views
    └── Returns response
```

This is convenient for development:
- Auto-reloads when you save a Python file
- Shows full error tracebacks in the browser
- No setup required

**Why it's not for production:**
- Single-threaded — one request at a time
- No process management — crashes are not recovered
- No HTTPS support
- Much slower than Nginx for static files
- Debug information exposed in errors

Django itself will warn you: *"This is a development server. Do not use it in a production setting."*

## The Database Layer

Every time your view calls something like `Article.objects.all()` or `Article.objects.get(pk=pk)`, Django's ORM translates it into a SQL query and sends it to the database server.

```
Django View
    │
    │  Article.objects.filter(published=True).order_by('-created_at')
    ▼
Django ORM
    │
    │  SELECT * FROM articles_article
    │  WHERE published = TRUE
    │  ORDER BY created_at DESC;
    ▼
Database Driver (psycopg2 for PostgreSQL)
    │
    │  TCP connection to localhost:5432
    ▼
PostgreSQL
    │
    │  Executes query, returns rows
    ▼
Django ORM
    │
    │  Wraps rows in Article model instances
    ▼
Your View has a QuerySet of Article objects
```

Database queries are almost always the biggest factor in view response time. A poorly optimised view might hit the database 50+ times for a single page load (the N+1 query problem). An optimised one hits it once or twice.

Django's debug toolbar shows exactly how many queries each view executed and how long they took — one of the most valuable tools during development.

## Key Terms

| Term | Definition |
|------|------------|
| **Static resource** | A file served as-is from disk — no computation required |
| **Dynamic resource** | A response built on request, based on data and logic |
| **Nginx** | A high-performance web server and reverse proxy |
| **Reverse proxy** | A server that forwards requests to another server on behalf of clients |
| **WSGI** | Web Server Gateway Interface — the standard interface between Python apps and web servers |
| **Gunicorn** | A Python WSGI HTTP server that manages worker processes |
| **ASGI** | Asynchronous Server Gateway Interface — the async successor to WSGI |
| **Worker process** | An independent Python process that handles one request at a time |
| **Middleware** | Code that wraps every request and response, applied in order |
| **URL dispatcher** | Django's system for matching URL paths to view functions |
| **ORM** | Object-Relational Mapper — Django's system for database access via Python objects |
| **N+1 problem** | A query pattern where fetching N objects triggers N additional queries |
| **runserver** | Django's built-in development server — not for production |

---

## Summary

- Servers distinguish **static** resources (served from disk) from **dynamic** resources (built per request) — Nginx handles static, Django handles dynamic
- The production stack is: **Nginx** → **Gunicorn** → **Django** → **PostgreSQL**
- **WSGI** is the Python standard interface; Gunicorn is the WSGI server that manages Django processes
- Django's request handling pipeline: **HttpRequest creation** → **middleware (request)** → **URL resolution** → **view execution** → **middleware (response)** → **HttpResponse**
- **Middleware** is code that runs on every request and response — it handles sessions, authentication, CSRF, security headers, and more
- Django's **development server** (`runserver`) is a convenience tool — single-threaded, auto-reloading, not for production
- Database queries inside views are the primary performance variable — the ORM translates Python into SQL
