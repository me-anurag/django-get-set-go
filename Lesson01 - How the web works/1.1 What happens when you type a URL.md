# 1.1 What Happens When You Type a URL

> Before you write a single line of Django, you need to understand the journey a request takes.
> Every framework, every bug, every deployment decision makes more sense once this is clear.

---

## The Moment You Press Enter

You type `https://www.google.com` into your browser and press Enter.

What looks like an instant response is actually a precise, multi-step chain of events happening in milliseconds. Let's walk through each one.

---

## Step 1 — The Browser Parses the URL

The browser doesn't see `https://www.google.com` as a single thing. It breaks it apart:

```
https   ://   www.google.com   /
  │              │               │
protocol       domain          path
```

| Part | What it means |
|------|---------------|
| `https` | The protocol — the agreed-upon language for this communication |
| `www.google.com` | The domain — the human-readable address of the server |
| `/` | The path — which specific resource you're asking for |

The browser now knows: *"I need to use HTTPS to talk to `www.google.com` and ask for `/`."*

---

## Step 2 — DNS Lookup (Finding the Server's Address)

The browser cannot connect to `www.google.com` directly. Computers communicate using **IP addresses** — numeric addresses like `142.250.80.46`. Domain names are just human-friendly aliases.

To find the IP, the browser performs a **DNS lookup**:

```
You type:          www.google.com
                        │
                  DNS Resolver
                  (your ISP or
                  8.8.8.8, etc.)
                        │
              Checks its own cache
              If not found → asks
              authoritative DNS
                        │
                  Returns:
               142.250.80.46
```

This lookup is fast — often under 20ms — and the result is cached so it doesn't repeat every time.

> **Think of it like this:** DNS is the phonebook of the internet. You look up a name (`google.com`) and get back a phone number (IP address) you can actually dial.

---

## Step 3 — TCP Connection (Opening a Channel)

Now the browser knows the IP. It needs to open a **reliable communication channel** to that address on **port 443** (the default port for HTTPS).

It does this using **TCP (Transmission Control Protocol)**, which uses a three-step handshake:

```
Browser                          Server
   │                                │
   │──── SYN (I want to connect) ──►│
   │                                │
   │◄─── SYN-ACK (OK, I'm ready) ───│
   │                                │
   │──── ACK (Great, let's go) ────►│
   │                                │
   Connection established ✓
```

This handshake ensures both sides are ready to communicate reliably before any data is sent.

---

## Step 4 — TLS Handshake (For HTTPS)

Since the URL starts with `https`, the browser and server also perform a **TLS handshake** — this is how encrypted communication is established.

In brief:
- The server proves its identity by showing a **certificate** (issued by a trusted authority)
- The browser verifies the certificate is legitimate
- Both sides agree on an **encryption key** to use for this session
- From this point, all data is encrypted

> You don't need to master cryptography. Just know that TLS is what makes the padlock appear in the browser and what prevents anyone from reading your data in transit.

---

## Step 5 — The HTTP Request is Sent

With the connection open and encrypted, the browser sends an **HTTP request**:

```
GET / HTTP/1.1
Host: www.google.com
User-Agent: Mozilla/5.0 ...
Accept: text/html
Accept-Language: en-US
```

This is a real, readable text message. It says:
- **GET** — I want to *retrieve* something (not submit, not delete)
- **/** — I want the root page
- **HTTP/1.1** — the version of HTTP I'm using
- **Host** — the domain (needed because one server can host many domains)

---

## Step 6 — The Server Processes the Request

The request travels across the internet to Google's servers. A server receives it and decides:

- What is being asked for?
- Does this user have permission?
- What should the response contain?

For a large site like Google, this involves many internal systems — routing, databases, caches, application logic. For a Django app, this is where your code runs: your URL patterns are matched, your view function executes, your database is queried.

The server then builds a response.

---

## Step 7 — The HTTP Response is Sent Back

The server sends a response back to the browser:

```
HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Content-Length: 14523

<!doctype html>
<html>
...
</html>
```

This contains:
- A **status code** (`200 OK` — success)
- **Headers** — metadata about the response
- A **body** — the actual HTML content

---

## Step 8 — The Browser Renders the Page

The browser receives the HTML and starts building the page:

1. Parses the HTML into a **DOM** (Document Object Model) — a tree of elements
2. Discovers linked resources: CSS files, JavaScript files, images
3. Sends **additional HTTP requests** to fetch those resources
4. Applies CSS to style the elements
5. Executes JavaScript
6. Paints the final result on screen

Each of those linked resources triggers its own mini version of this entire process.

---

## The Full Picture

```
[You press Enter]
       │
       ▼
[Browser parses the URL]
       │
       ▼
[DNS lookup: domain → IP address]
       │
       ▼
[TCP handshake: connection established]
       │
       ▼
[TLS handshake: encryption established] ← only for HTTPS
       │
       ▼
[HTTP request sent to server]
       │
       ▼
[Server processes request, runs application code]
       │
       ▼
[HTTP response sent back]
       │
       ▼
[Browser renders the page]
       │
       ▼
[You see the result]
```

---

## Why This Matters for Django

Every Django application lives at **Step 6** — the server processing the request.

When you write a Django view, you are writing the code that runs when the server receives a request and needs to build a response. When you define URL patterns, you are telling Django how to match the path from the request to the right piece of code.

Understanding this flow means you'll never be confused about:
- Why URLs matter and how they map to your code
- What a "request object" is in Django (it's the HTTP request, represented as a Python object)
- What a "response" is (the HTTP response your view returns)
- Why Django needs a web server in front of it for production

---

## Key Terms

| Term | Definition |
|------|------------|
| **URL** | Uniform Resource Locator — the full address used to locate a resource |
| **Protocol** | A set of rules two parties agree on for communication |
| **DNS** | Domain Name System — translates domain names to IP addresses |
| **IP Address** | A numeric identifier for a device on a network |
| **TCP** | Transmission Control Protocol — a reliable, connection-based transport protocol |
| **TLS** | Transport Layer Security — provides encryption for HTTPS connections |
| **HTTP Request** | A message from client to server asking for or submitting something |
| **HTTP Response** | The server's reply, containing status, headers, and usually a body |
| **Rendering** | The browser's process of turning HTML/CSS/JS into a visible page |

---

## Summary

When you type a URL and press Enter:

1. The browser parses the URL into protocol, domain, and path
2. DNS resolves the domain to an IP address
3. A TCP connection is opened to the server
4. For HTTPS, a TLS handshake secures the channel
5. The browser sends an HTTP request
6. The server runs application code and builds a response
7. The browser receives the response
8. The browser renders the HTML into a visible page

This entire journey — from keypress to visible page — is the foundation of everything Django does.
